# PASSMAP Transition Scoring QA Framework

## 목적
- PASSMAP의 직무/산업 전환 scoring 품질을 반복 검증하기 위한 운영 기준을 정의한다.
- 동일 케이스 중복 실행을 줄이고, 라운드 간 비교 기준을 고정한다.
- false high / false low 패턴과 calibration 판단을 추적 가능하게 유지한다.

## 언제 이 문서를 쓰는가
- 새 QA 라운드를 설계하거나 실행 범위를 정할 때 본다.
- 새 케이스를 gold set에 추가하기 전에 평가 기준을 확인할 때 본다.
- scoring 보정 제안을 하기 전에 어떤 오류가 치명 오류인지 확인할 때 본다.

## 다른 문서와의 관계
- 케이스 자산은 `Transition_Scoring_Gold_Set.md`에서 관리한다.
- 라운드 실행 결과는 `05_Execution/Accuracy_QA_Log.md`에 남긴다.
- scoring 조정 결정은 `05_Execution/Scoring_Calibration_Log.md`에 남긴다.

## 왜 절대점수보다 등급/순위/치명오류를 먼저 보는가
- 점수 숫자는 세부 구현 변화에 흔들리기 쉽지만, 등급 밴드와 상대 순위는 사용자 체감 품질과 더 직접적으로 연결된다.
- 실제 실패는 `숫자 3점 차이`보다 `붙으면 안 되는 케이스를 높게 준 false high`에서 더 크게 발생한다.
- 따라서 1차 판정은 등급 밴드, 후보 간 순위, 치명 오류 여부로 하고, 절대점수는 2차 보조 지표로 본다.

## 평가 대상 범위
- 직무 유사도
- 산업 유사도
- 책임/역할 범위 차이
- evidence 충족도
- 전환 리스크

## 평가 관점
- 구조 정확도
  - 직무/산업/역할 범위/evidence/리스크가 빠짐없이 읽혔는지 본다.
- 판정 정확도
  - 최종 등급과 점수 밴드가 전문가 기대 범위 안에 들어오는지 본다.
- 순위 정확도
  - 같은 라운드 안에서 더 적합한 케이스가 덜 적합한 케이스보다 안정적으로 높게 나오는지 본다.
- 설명-점수 정합성
  - 설명이 강조하는 강점과 약점이 실제 점수 방향과 충돌하지 않는지 본다.

## 치명 오류 정의
- false high
  - 전문가 기준으로 `낮아야 하는 전환`을 높은 밴드로 준 경우
- false low
  - 전문가 기준으로 `가능성이 있는 전환`을 과도하게 낮게 준 경우
- evidence 무시형 점수
  - 핵심 증거가 있는데 점수에 반영되지 않거나, 핵심 증거가 없는데 과대평가한 경우
- explanation mismatch
  - 설명은 보수적인데 점수는 높거나, 설명은 긍정적인데 점수는 낮은 경우

## 점수 밴드 정의
| 밴드 | 기준 |
|---|---|
| 매우 높음 | 전환 설득력이 매우 높고 치명 리스크가 거의 없음 |
| 높음 | 강한 전환 근거가 있고 보완 포인트가 제한적임 |
| 중간 | 가능성은 있으나 역할/산업/evidence 측면에서 확인 필요 |
| 낮음 | 전환 저항이 크고 추가 근거 없이는 설득력이 약함 |
| 매우 낮음 | 현재 정보 기준으로 false high를 피해야 하는 구간 |

## QA 라운드 운영 방식
1. 라운드 ID를 먼저 발급한다.
2. gold set에서 이번 라운드 대상 케이스를 고른다.
3. 각 케이스의 기대 등급과 기대 점수 범위를 기준으로 결과를 비교한다.
4. hit rate, one-band 허용치, ranking agreement, 설명 정합성을 기록한다.
5. 치명 오류가 있으면 `Accuracy_QA_Log.md`에 남기고, 조정이 필요하면 `Scoring_Calibration_Log.md`로 넘긴다.

## 중복 방지 규칙
- 이미 등록된 `case_id`는 재사용하되 의미를 바꾸지 않는다.
- 새 케이스가 기존 케이스와 현재 직무/산업, 목표 직무/산업, 핵심 전환 이유가 실질적으로 같으면 신규 등록하지 않는다.
- 동일 케이스를 다시 실행하는 경우는 새 row를 만들지 않고 새 round log에만 결과를 추가한다.

## 케이스 ID 규칙
- 형식: `TSQ-CASE-0001`
- `TSQ`는 transition scoring QA를 뜻한다.
- 숫자는 append-only로 증가시키며, 기존 ID를 재할당하지 않는다.

## 라운드 ID 규칙
- 형식: `TSQ-RND-20260402-01`
- 날짜 + 일자 내 순번 구조를 사용한다.
- 실험성 재실행도 별도 round ID를 발급한다.

## Future Update Rule
- 평가 축이나 점수 밴드를 바꿀 때는 기존 라운드와 비교 가능성이 깨지는지 먼저 검토한다.
- 비교 기준이 바뀌면 framework 문서에 변경 이유를 남기고, calibration log에 연결 근거를 남긴다.
- 기존 케이스 기대값을 조정할 때는 조용히 덮어쓰지 말고 gold set notes와 calibration log 둘 다 갱신한다.

# PASSMAP 전환 5축 점수 QA Framework (2026-04-03 Baseline)

## 1. 이번 라운드 목적
- 기존 축과 완전히 달라졌기 때문에, 과거 QA 기준/테스트 러너를 그대로 신뢰하지 않는다.
- 먼저 문서 기준을 SSOT로 확정한다.
- 우선 테스트 범위는 Axis 1 / Axis 4 / Axis 5만 시작한다.
- Axis 2 / Axis 3는 이후 라운드에서 별도 설계/검증한다.
- 이 문서가 이후 테스트 러너, 골드셋, calibration 논의의 기준점이다.

## 2. 새 5축 정의
1. 직무 구조 연결성
2. 산업 맥락 연결성
3. 역할 범위 연결성
4. 고객 유형 연결성
5. 직무 성격 연결성

## 3. 이번 1차 테스트 범위
- Axis 1: 직무 구조 연결성
- Axis 4: 고객 유형 연결성
- Axis 5: 직무 성격 연결성
- 이번 1차 테스트는 5축 전체 완성 검증이 아니라, 새 체계에서 가장 해석 충돌이 잦을 가능성이 높은 축부터 baseline을 세우는 목적이다.
- 따라서 이번 라운드의 성공 기준은 코드 구현 완료가 아니라 판정 기준, 예시, 테스트 분류 원칙이 문서로 명확히 잠기는 것이다.

## 6. 축별 설명 의도 + 1~5점 정의

### **Axis 1. 직무 구조 연결성**
설명 의도:
현재 직무와 목표 직무가 하는 일 자체가 얼마나 비슷한가

1점: 지금까지 해온 직무와 지원 직무의 핵심 업무가 많이 다릅니다.
2점: 지금까지 해온 직무와 지원 직무의 핵심 업무가 상당 부분 다릅니다.
3점: 기본적으로는 핵심 업무가 연결되지만, 어떤 점이 이어지는지 더 풀어주면 좋습니다.
4점: 해오신 일과 지원 직무가 꽤 자연스럽게 이어집니다.
5점: 지금까지의 경험을 거의 그대로 강점으로 연결할 수 있습니다.

### Axis 2. 산업 맥락 연결성
설명 의도:
일하던 업계와 지원 업계의 배경 차이

1점: 산업의 결이 매우 다릅니다. 산업의 차이가 중요한 직무라면 설명이 필요합니다.
2점: 산업의 결이 다릅니다. 산업의 차이가 중요한 직무라면 설명이 필요합니다.
3점: 산업의 큰 차이는 없지만, 업계 이해를 보여주면 좋습니다.
4점: 산업의 배경이 꽤 비슷해서 경험을 연결하기 좋습니다.
5점: 산업의 배경이 매우 유사하여 자연스럽게 이어집니다.

### Axis 3. 역할 범위 연결성
설명 의도:
내가 맡아본 책임의 깊이/범위와 지원 직무의 기대 수준이 맞는가

1점: 지금까지 맡아온 책임 수준과 지원 직무의 기대 수준 차이가 큽니다.
2점: 일부 연결되는 경험은 있지만, 책임 범위 차이를 설명해줘야 합니다.
3점: 기본 연결은 되지만, 어디까지 맡아봤는지 더 구체적으로 보여줄 필요가 있습니다.
4점: 맡아온 책임 수준이 지원 직무와 꽤 잘 맞습니다.
5점: 책임 범위와 깊이가 매우 비슷해 바로 강점으로 가져가기 좋습니다.

### **Axis 4. 고객 유형 연결성**
설명 의도:
B2B/B2C, 내부고객/외부고객 등 상대 대상의 차이

1점: 지금까지 상대해온 고객과 지원 직무의 대상이 많이 다릅니다.
2점: 일부 비슷하지만, 고객 특성 차이를 보완해서 설명하는 게 좋습니다.
3점: 큰 틀에서는 이어지지만, 고객 차이를 짚어주면 더 설득력 있어집니다.
4점: 상대해온 고객 유형이 비슷해 사례를 연결하기 좋습니다.
5점: 고객 유형이 매우 비슷해 기존 경험을 그대로 활용하기 좋습니다.

### **Axis 5. 직무 성격 연결성**
설명 의도:
조율형/분석형/운영형/기획형 등 일의 결이 비슷한가

1점: 일하는 방식과 문제를 푸는 방식이 많이 다릅니다.
2점: 비슷한 점도 있지만, 일하는 방식의 차이를 설명해주는 게 좋습니다.
3점: 어느 정도는 맞지만, 어떤 방식으로 일해왔는지 더 보여주면 좋습니다.
4점: 일하는 방식이 꽤 비슷해서 적응 부담이 크지 않습니다.
5점: 일하는 방식이 매우 비슷해 자연스럽게 연결됩니다.

## 7. 이번 1차 테스트에서 특히 확인할 것
- Axis 1은 직무명이 비슷해 보여도 실제 업무 구조가 다르면 낮아질 수 있음을 검증해야 한다.
- Axis 4는 같은 직무군이라도 고객군 차이가 크면 설득 구조가 달라진다를 검증해야 한다.
- Axis 5는 겉직무보다 일의 결/사고방식/문제 해결 방식의 유사성을 검증해야 한다.
- 즉, 이번 라운드는 단순 유사 직무 찾기가 아니라 구조 / 고객 / 일의 성격을 분리해서 읽는 축의 유효성을 테스트하는 것이다.

## 8. 테스트 케이스 설계 원칙
- 골드셋은 점수 맞추기용 숫자놀이가 아니라 왜 그 축이 그렇게 읽혀야 하는지가 먼저 서야 한다.
- 같은 대분류 안 이동도 낮아질 수 있다.
- 다른 대분류 이동도 높아질 수 있다.
- 직무명 표면 유사도만으로 점수를 주지 않는다.
- 산업 차이는 이번 1차 테스트의 핵심이 아니므로 Axis 1/4/5 판정에 불필요하게 과도 반영하지 않는다.
- 한 케이스마다 최소한 아래를 기록한다.
  - current job
  - target job
  - current industry
  - target industry
  - axis 1 expected band / reasoning
  - axis 4 expected band / reasoning
  - axis 5 expected band / reasoning
  - disagreement risk memo
- expected exact score를 강제하지 말고, 초반에는 expected band(예: low / mid / high) + 이유 중심으로 설계해도 된다는 원칙을 명시한다.

## Per-Axis Reasoning Minimum Fields
- 모든 축의 reasoning에는 최소한 아래 2요소가 들어가야 한다.
  - 해당 축의 연결성이 높은 이유 또는 낮은 이유
  - 그 판단의 근거가 되는 구체적 관찰 포인트

- Axis 1 (직무 구조 연결성) 최소 reasoning 포인트
  - 핵심 업무의 유사성
  - 현재 직무와 목표 직무의 역할 구조 차이
  - 수행해온 일의 본질이 직접 이어지는지

- Axis 2 (산업 맥락 연결성) 최소 reasoning 포인트
  - 업계 배경 차이
  - 산업 특유의 문맥/도메인 이해 필요 여부
  - 산업 결 차이가 설득 부담으로 작동하는지

- Axis 3 (역할 범위 연결성) 최소 reasoning 포인트
  - 책임 깊이
  - 책임 범위
  - 기대 레벨과 실제 수행 범위의 차이
  - 오너십 수준

- Axis 4 (고객 유형 연결성) 최소 reasoning 포인트
  - 고객 구조
  - customer market (B2B / B2C / 내부고객 / 외부고객)
  - buying motion 또는 고객 특성 차이

- Axis 5 (직무 성격 연결성) 최소 reasoning 포인트
  - 일의 결
  - 조율형 / 분석형 / 운영형 / 기획형
  - mission type
  - role weight
  - 문제를 푸는 방식 / 일하는 방식의 유사성

## Band Judgment Recording Rule
- exact score보다 `band`를 우선 기록한다.
- reasoning 없이 `band`만 기록하는 케이스는 금지한다.
- disagreement가 발생하면 반드시 이유를 남긴다.
- disagreement가 발생하면 follow-up 방향도 함께 기록한다.

## 2026-04-04 5축 전체 baseline 확장 메모
- 기존 문서 잠금 취지는 유지하되, 이제 1차 gold set baseline 판독은 Axis 1 / Axis 4 / Axis 5 한정이 아니라 Axis 1~5 전체 기준으로 읽는다.
- 이는 과거 1 / 4 / 5 우선 잠금 판단을 뒤집는 것이 아니라, Axis 2 / Axis 3 정의까지 문서 SSOT가 잠긴 뒤 baseline 해석 범위를 확장한 운영 메모다.
- gold set 케이스는 표면 직무명 유사도만 보지 않고 아래 5개 판단 층위를 분리해서 기록한다.
  - 직무 구조 연결성
  - 문제 해결 방식 / 업무 수행 메커니즘
  - 산출물 / 평가 기준 / 기대 결과물
  - 산업 맥락 / 고객 / 이해관계자 구조
  - 역할 범위 / 책임 수준 / 오너십 범위
- 이번 라운드의 expected는 exact score가 아니라 `band + reasoning` 기준으로만 적재한다.
- runner 구현 전까지는 사람이 읽어도 같은 방향의 band 판단이 나오는지 먼저 확인하는 것을 baseline 목적에 둔다.

## PASSMAP Job / Industry Input SSOT Rule
- gold set에 쓰는 `currentJob`, `targetJob`, `currentIndustry`, `targetIndustry`는 본 문서의 직무/산업 SSOT 명칭만 사용한다.
- 임의 축약, 외부 taxonomy, 혼합 라벨, 해석용 별칭은 금지한다.
- 직무/산업의 세부 맥락은 label을 변형하지 말고 `reasoningHints`에서 설명한다.
- 같은 계열 전환이라도 표면 유사도 때문에 과대평가하지 않도록 구조 / 메커니즘 / 산출물 / 산업 맥락 / 책임 범위를 분리 기록한다.
- PASSMAP 입력 SSOT 외 taxonomy가 섞인 케이스는 baseline 후보에서 제외한다.
