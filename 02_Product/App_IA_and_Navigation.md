# App IA and Navigation

## 문서 목적
내일 미팅에서 PASSMAP를 단일 실험 화면이 아니라 하나의 서비스 구조로 설명하기 위한 앱 정보구조 문서.

현재 구현 화면과 장기 제품 방향을 구분해서, 어떤 메뉴와 흐름으로 서비스가 보일지를 정리한다.

## 홈 우선 제작 기준
- 홈은 실제 첫 제작 우선순위 화면이다.
- 홈은 이후 업무 업데이트, 분석 리포트, JD 준비도, 이력서 관리로 이동하는 연결 허브다.
- 따라서 IA 설명에서도 홈을 단순 소개 화면이 아니라 실제 운영 진입점으로 취급한다.

## UI Structure Decisions

### 업무 관리 화면 판단
- `업무 관리`는 운영 숫자판이 아니라 `오늘 해야 할 행동`, `최근 반영 상태`, `시간 흐름`을 보는 화면으로 정의한다.
- 상단 정보는 `오늘 해야 할 일`, `이번 주 기록 상태`, `최근 반영 상태` 같은 행동/시간 기준으로 재구성한다.
- `분석 리포트 수`, `JD 비교 수` 같은 숫자 요약은 보조 정보로 내리고 주연 정보로 두지 않는다.
- 이 화면의 핵심 UI는 달력, 주간 기록 상태, 최근 반영 상태다.

### 이력서 쉽게 업데이트 하기 화면 판단
- `이력서 쉽게 업데이트 하기`는 단일 입력폼이 아니라 `오늘 기록 -> 이번 주 기록 -> 해석 결과 -> 목표 준비도` 흐름으로 설계한다.
- 입력 단위는 반드시 `오늘 기록`, `이번 주 기록`으로 분리한다.
- 우측 설명 박스를 늘리는 방식보다 달력과 상태 요약을 앞세워 기록 흐름을 먼저 보이게 한다.

### 시간 축 판단
- PASSMAP은 분석 서비스만이 아니라 기록 서비스 성격이 강하므로 시간 흐름이 보여야 제품 본질이 드러난다.
- 달력은 장식용 보조 UI가 아니라 기록이 누적되고 반영되는 과정을 보여주는 핵심 구조다.
- 따라서 `업무 관리`, `이력서 쉽게 업데이트 하기` 두 화면 모두 날짜 기반 상태 UI를 기본 전제로 둔다.

## 2026-04-02 Sidebar IA / Resume Flow Update

### 확정된 IA 기준
- 사이드바에 `현재 기준 직무` 선택 UI를 두고, 설정 안에 숨기지 않는다.
- 현재 기준 직무는 기록/해석이 어떤 문맥 위에서 쌓이는지를 보여주는 persistent context다.
- JOB shell 기준 메뉴 구조는 아래 순서를 기준으로 삼는다.
  - 홈
  - 업무 관리
  - 이력서 쉽게 업데이트 하기
  - 해석 결과
  - 연결 포인트
  - 설정
- settings는 shell 수준 1차 구조를 갖고, 그 안에 알림 설정 shell을 둔다.
- 알림 설정은 실제 발송 로직이 아니라 entry/shell 수준으로만 기록한다.

### 현재 기준 직무 선택 위치와 의미
- 현재 기준 직무는 설정 내부에만 숨기는 값이 아니라 기록 / 해석 / 연결 포인트의 기준 문맥이다.
- 기존 혼합 라벨 예시인 `프로덕트(기획/전략)`은 PASSMAP 표준 분류와 불일치하므로 장기 기준으로 유지하지 않는다.
- 목표 방향은 PASSMAP 직무 taxonomy 기준 2단 선택이다.
  - 대분류 선택
  - 중분류 선택
  - 표시값은 `대분류 > 중분류` 또는 2줄형
- 이 항목은 중요 설계 판단이며, fully wired final state로 기록하지 않는다.

### resume 입력 흐름 기준 변경
- `src/components/mvp/PmRecordInput.jsx`는 큰 섹션을 숨기는 폼이 아니라, 섹션 내부 chip을 편집하는 입력 구조를 기준으로 삼는다.
- 유지되는 섹션
  - 업무 유형
  - 협업 맥락
  - 결과/후속 메모
- 각 섹션은 preset chip을 초기값으로 가지되, 이번 기록 세션 기준으로 삭제/추가 가능해야 한다.
- textarea는 상단 시작점이 아니라 마지막 정리 단계로 둔다.
- 사용자 의도는 입력 구조 제거가 아니라 입력 재료의 유연한 편집이라는 점을 이후 유사 작업의 기준 판단으로 남긴다.

### 캘린더 연동 기준
- `[이력서 쉽게 업데이트 하기]` 캘린더는 별도 패널을 계속 늘리는 방향이 아니라, work 쪽 기준 컴포넌트를 재사용하는 방향으로 맞춘다.
- 기준 컴포넌트는 `src/components/home/RecordCalendarCard.jsx`다.
- 목표는 사용자가 work/resume을 다른 시스템이 아니라 같은 기록 흐름의 다른 뷰로 느끼게 하는 것이다.
- 범례 / 월간 캘린더 / 표시 규칙은 같은 의미를 화면마다 따로 만들지 않는 것을 원칙으로 한다.
- 다만 resume 화면은 오른쪽 패널 폭에 맞는 compact density tuning이 별도 과제로 남아 있다.
