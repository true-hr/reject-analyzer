# Home Dashboard Design Spec

## 문서 목적
PASSMAP 홈 화면을 실제 첫 제작 대상으로 삼기 위한 상세 설계 기준 문서.

이 문서는 홈 화면의 역할, 섹션, 카드, 버튼, 연결 흐름, 검증 기준을 먼저 고정해서 이후 Codex, Claude, ChatGPT가 같은 기준으로 작업할 수 있게 만드는 데 목적이 있다.

## 홈을 먼저 만드는 이유
- 홈은 PASSMAP의 현재 상태를 한눈에 보여주는 상태 허브다.
- 홈은 최근 활동과 성과를 묶어 보여주는 활동 허브다.
- 홈은 사용자가 다음에 해야 할 일을 제안하는 다음 액션 허브다.
- 첫 화면이 홈으로 고정되어야 이후 업무 업데이트, 분석 리포트, JD 준비도, 이력서 관리 화면을 자연스럽게 연결할 수 있다.

## 화면 역할 고정
- 상태 허브: KPI, 현재 진행 현황, 최근 분석 상태를 집약한다.
- 활동 허브: 최근 업무 업데이트와 최근 분석 리포트를 시간 순서로 보여준다.
- 다음 액션 허브: 추천 액션과 빠른 이동 버튼으로 다음 행동을 유도한다.

## 레이아웃
- 좌측 사이드바
  - 홈
  - 업무 업데이트
  - 분석 리포트
  - JD 준비도
  - 이력서 관리
  - 목표 설정
  - 내 프로필
  - 액션 가이드
- 상단 헤더
  - 페이지 타이틀: `PASSMAP Home`
  - 보조 문장: `오늘의 상태와 다음 액션을 한 번에 확인합니다.`
  - 우측 CTA: `업데이트 등록`, `분석 시작`
- 본문 메인 그리드
  - 1행: KPI 카드 4개
  - 2행: 현황 패널, 성과 패널
  - 3행: 최근 업무 업데이트, 최근 분석 리포트
  - 4행: 추천 액션, 활동 추이 그래프

## KPI 카드 4개
| 카드명 | 샘플 수치 | 보조 문구 | 클릭 이동 |
| --- | --- | --- | --- |
| 이번 주 업데이트 | 6건 | 지난주 대비 +2 | 업무 업데이트 |
| 완료된 분석 | 14건 | 최근 30일 기준 | 분석 리포트 |
| JD 준비도 평균 | 78점 | 상위 목표 JD 기준 | JD 준비도 |
| 이력서 최신화율 | 82% | 최근 수정 7일 이내 | 이력서 관리 |

## 현황 패널
- 패널명: `현재 진행 현황`
- 표시 항목
  - 진행 중 목표: `프로덕트 매니저 전환`
  - 최근 등록 업데이트: `고객 VOC 분류 자동화 프로젝트`
  - 최근 분석 상태: `데이터 기반 기획 역량 상승`
  - 이번 주 핵심 과제: `JD 2건 비교 분석 완료`
- 버튼
  - `목표 확인`
  - `업데이트 이어서 작성`

## 성과 패널
- 패널명: `이번 달 성과`
- 표시 항목
  - 누적 업데이트: `21건`
  - 누적 분석 리포트: `8건`
  - 목표 적합도 최고점: `84점`
  - 추천 액션 완료율: `67%`
- 버튼
  - `성과 상세 보기`
  - `리포트 모아보기`

## 최근 업무 업데이트 패널
- 패널명: `최근 업무 업데이트`
- 리스트 예시
  - `VOC 태깅 기준 정리 완료` / `오늘 09:10`
  - `신규 온보딩 플로우 개선안 작성` / `어제 18:20`
  - `주간 성과 리포트 템플릿 정비` / `3월 31일`
- 각 행 액션
  - `보기`
  - `수정`
  - `분석 연결`

## 최근 분석 리포트 패널
- 패널명: `최근 분석 리포트`
- 리스트 예시
  - `B2B PM JD 적합도 분석` / `적합도 84`
  - `서비스 기획 JD 격차 분석` / `적합도 76`
  - `운영 기획 역할 전환 분석` / `적합도 72`
- 각 행 액션
  - `리포트 보기`
  - `액션 보기`

## 추천 액션 패널
- 패널명: `추천 액션`
- 카드 예시
  - `JD 준비도 점수가 낮은 항목부터 보강하세요.`
  - `최근 업데이트 2건을 리포트에 연결하면 적합도 해석이 더 정확해집니다.`
  - `이력서 최신화율을 90% 이상으로 올리면 제출 준비 상태가 안정됩니다.`
- 버튼 체계
  - 기본 CTA: `지금 하기`
  - 보조 CTA: `나중에 보기`

## 활동 추이 그래프
- 패널명: `활동 추이`
- 그래프 구조
  - X축: 최근 8주
  - Y축 1: 업데이트 등록 수
  - Y축 2: 분석 완료 수
- 샘플 데이터
  - 1주차 `2 / 1`
  - 2주차 `3 / 1`
  - 3주차 `1 / 2`
  - 4주차 `4 / 2`
  - 5주차 `2 / 1`
  - 6주차 `5 / 3`
  - 7주차 `1 / 1`
  - 8주차 `3 / 2`

## 연결 흐름
- 홈 KPI 카드 클릭 -> 해당 기능 화면으로 이동
- 최근 업무 업데이트의 `분석 연결` 클릭 -> 분석 리포트 생성 흐름 시작
- 최근 분석 리포트의 `액션 보기` 클릭 -> 액션 가이드 또는 후속 작업 안내
- 추천 액션의 `지금 하기` 클릭 -> 관련 화면으로 즉시 이동

## 검증 기준
- 홈만 열어도 상태, 최근 활동, 다음 행동이 모두 이해되어야 한다.
- 각 카드와 패널은 실제 다음 이동 화면과 연결 문장이 있어야 한다.
- 미팅 데모 설명 시 `운영형 제품 인상`을 줄 수 있어야 한다.
- 이후 구현 브리프로 바로 사용할 수 있을 만큼 명칭과 샘플 데이터가 고정되어 있어야 한다.

## 데모 설명 문장
`PASSMAP 홈은 단순 요약 대시보드가 아니라, 현재 상태 확인과 최근 활동 파악, 다음 액션 실행까지 한 화면에서 이어지는 운영형 허브 화면이다.`

## JOB Shell-Level Rail 반영 메모
- JOB에서는 centered content 내부 카드형 사이드바 접근을 더 이상 채택하지 않는다.
- 현재 기준 구조는 Shell-level left rail + right content 조합으로 고정한다.
- JOB view switch는 아래 3개로 운영한다.
  - `analysis`
  - `work`
  - `resume`
- `analysis`는 hero, 설명, 포인트 카드, CTA 중심의 landing / entry 성격이다.
- `work`(`HomeDashboard`)는 여러 블록을 넓게 쓰는 dashboard 성격이다.
- 따라서 `analysis`와 `work`는 동일 width policy를 쓰지 않는다.
- `resume`은 현 구조를 유지하되, 필요 시 중간 폭 정책을 별도로 검토한다.

## Work 화면 월간 캘린더 반영
- `work` 화면 역할은 단순 상태판이 아니라 업무 기록 -> 해석 -> 자산화가 보이는 운영 대시보드다.
- 기존 1주 strip 형태 기록 캘린더는 폐기하고, 날짜 단위로 업무를 읽을 수 있는 월간 캘린더를 채택했다.
- 캘린더 범례는 시스템 단계가 아니라 실제 업무 유형 기준으로 운영한다.
  - `문의 대응`
  - `이슈 조율`
  - `문서/보고`
  - `운영 개선`
- 선택 날짜를 누르면 해당 날짜의 업무 목록, 요약 문장, 반영 문장, 강점 태그, 보완 포인트를 보여주는 자산화 상세 카드가 갱신된다.
- 캘린더 주변에는 이번 주 / 오늘 요약과 함께 이번 달 자산 요약을 배치해 기록 흐름과 자산화 결과를 같이 읽게 한다.

## 데이터 계약 메모
- `work` 캘린더의 source of truth는 `records` 중심 normalized contract다.
- 각 record는 최소 아래 축을 유지한다.
  - `id`
  - `date`
  - `source`
  - `workType`
  - `title`
  - `summary`
  - `reflectedSentence?`
  - `strengthTags?`
  - `linkedAssetIds?`
- 날짜 상세, 이번 주 / 오늘 요약, 이번 달 자산 요약은 하드코딩 문장이 아니라 record 배열에서 파생값으로 계산한다.
- 이 구조는 추후 Notion DB row, Google Calendar event adapter를 붙일 때도 `날짜 / 업무 유형 / 요약 텍스트` 3축을 유지하는 것을 전제로 한다.
- 화면 역할은 계속 분리한다.
  - `analysis` = entry / landing
  - `work` = dashboard
  - `resume` = asset review
  - width policy는 view별 성격에 맞게 분리한다.

## Work Record Multi-Job 확장 메모
- 현재 PM 입력 taxonomy owner는 `src/components/mvp/PmRecordInput.jsx`다.
- 현재 구조는 공통 입력 shell 위에 PM 전용 `workTags`, placeholder, sampleRecord가 하드코딩된 상태다.
- multi-job 확장 시에는 공통 3축을 유지한다.
  - 업무 유형
  - 협업 맥락
  - 결과/후속 메모
- job ontology는 계속 해석용 공통 자산으로 유지한다.
- 입력 taxonomy는 별도 `workRecord` data layer로 분리한다.
- 연결 계약은 `job id -> input preset registry` 구조를 사용한다.
- `work` dashboard에서 보이는 자산화 결과 구조는 최대한 공통으로 유지하고, 직무별 입력 preset과 일부 copy만 바꾼다.

## 2026-04-02 Home Dashboard Direction Update

### 업무 관리 홈 요약 카드 방향 확정
- 기존 `기록 수 / 반영 자산 / 보완 포인트` 중심 카드는 사용자 체감이 약해서 폐기한다.
- 홈 상단 요약 카드는 아래 4개 방향을 기준으로 삼는다.
  - 오늘 기록 상태
  - 이번 주 해낸 일
  - 이번 주 만든 이력서 문장
  - 쌓이고 있는 커리어 방향
- 무의미한 숫자보다 실제 업무 단위 숫자, 이력서 문장 연결, 최근 누적 커리어 방향을 보여주는 쪽을 우선한다.
- 홈은 관리 대시보드보다 `이력서 업데이트가 편해지고 커리어가 쌓이는 느낌`이 먼저 와야 한다.

### 상단 CTA 위계 판단
- `오늘 기록하기 / 반영 문장 보기 / 연결 포인트 보기` 대형 CTA 3카드는 중요도 대비 공간 점유가 크다.
- 이 영역은 메인 hero처럼 보이면 안 된다.
- 기준 판단은 `secondary quick actions`로 축소/재배치하는 것이다.
- 화면의 메인 위계는 결과 확인 카드와 기록 흐름이 먼저 와야 한다.
- 이 항목은 설계 방향 확정이며, 브라우저 최종 UX 검증 완료로 기록하지 않는다.

### 캘린더 표현 기준
- 홈 달력은 `src/components/home/HomeDashboard.jsx`의 기록 흐름을 기준으로 삼는다.
- 같은 의미의 캘린더를 화면마다 따로 만들지 않고, 범례/월간 캘린더/표시 규칙을 공용화하는 방향을 유지한다.

## Resume 입력 화면 재정의
- `[이력서 쉽게 업데이트 하기]`는 결과를 풍성하게 보여주는 화면보다 빠르게 기록하고 이력서로 이어지게 만드는 입력 화면이어야 한다.
- 현재 문제는 아래처럼 정리한다.
  - 칩 섹션 3개가 같은 패턴으로 반복되어 세로 높이를 많이 차지함
  - 오른쪽 보조 패널 비중이 큼
  - 공간 효율이 낮고 답답해 보임
- 새 기준은 아래와 같다.
  - 왼쪽 = 빠른 입력
  - 오른쪽 = compact 보조 캘린더 / 선택 날짜 요약
  - 업무 관리와 같은 디자인 언어는 유지하되 정보 밀도와 면적 배분은 다르게 가져간다

## 캘린더 공용화와 Compact Density 원칙
- `[이력서 쉽게 업데이트 하기]` 초기 캘린더는 별도 패널 / 별도 mock 성격이 강했다.
- 분리된 시스템처럼 보이지 않도록 `src/components/home/RecordCalendarCard.jsx` 공용 기준을 채택했다.
- 공용화 후 resume 화면 오른쪽 패널 폭에 비해 셀 내부 정보가 과해 UI가 깨져 보이는 문제가 있었다.
- 핵심 판단은 아래와 같다.
  - 공용화 자체가 실패한 것이 아니다
  - 공용화 후 density tuning이 빠진 상태였다
- 수정 원칙은 아래와 같다.
  - 셀 내부는 요약형
  - 상세 텍스트는 선택 날짜 상세 영역으로 이동
  - 필요 시 오른쪽 패널 compact variant / 폭 조정

## 멘트/라벨 중복 제거 원칙
- 섹션 제목과 내부 라벨이 같은 의미를 반복하면 폼이 무겁고 복잡하게 느껴진다.
- 같은 의미의 라벨/설명 중복은 줄이고, 각 섹션은 한 번만 또렷하게 읽히게 정리하는 방향을 채택한다.
