  - "우선순위를 정하고" (양쪽에 반복 등장)
  - "개발·디자인 협업" (기반으로 공통 언급)

- 겹쳐도 되는 공통 표현:
  - "서비스기획 경험은 연결성이 높다"는 기본 진입 문장은 동일 출발점이므로 허용됨
  - "IA, 화면설계, 기능정의" 열거는 현재직 설명으로 양쪽 공유 가능
- 겹치면 안 되는 위험 표현:
  - PM V2: "사용자 문제를 정의하고 우선순위를 정한다" → PO V1에도 "우선순위를 정하고" 등장

- 실제 오염 위험:

### 추가 검토 결과

**Axis1에서 PM/PO 차이가 살아 있는가:**
서비스기획→PM V2의 Axis1은 "제품이 풀어야 할 문제를 정하고 우선순위·성과 확인"에 집중.

→ 살아 있음. PASS.

**Axis3에서 서비스기획→PM과 서비스기획→PO가 구분되는가:**
PM V2 Axis3: "문제 정의 ~ 우선순위 ~ 출시 후 성과"까지의 오너십 범위.
PO V1 Axis3: "기획 산출물 중심 vs 제품팀 실행 단위와 출시 흐름 관리".
→ 의미 구분은 되지만 논리 패턴(산출물 중심 → 실행 관리)이 유사함. 단, 타깃이 PM 오너십과 PO 실행 책임으로 확실히 달라 허용 범위 내.

**데이터분석→PM이 퍼포먼스마케팅→PM처럼 보이지 않는가:**
본문은 충분히 다름. DA: 분석 인사이트 → 제품 결정. PerfMkt: 광고 성과 → 제품 퍼널.

**퍼포먼스마케팅→PM이 단순 데이터분석→PM처럼 보이지 않는가:**
PerfMkt는 그로스 PM, Growth PM 연결어, 랜딩페이지, ROAS, 세그먼트, 광고 채널 언어를 사용하여 DA와 구분됨.
→ 안전.

**운영기획→PM이 서비스기획→PM처럼 보이지 않는가:**
운영→PM은 운영 자동화, 어드민, 백오피스, 처리시간 단축, CS 툴 같은 운영 특유 언어를 사용. 서비스기획→PM은 IA, 화면설계, 정책서, 기획 산출물 언어를 사용.
→ 완전히 다름. 안전.
