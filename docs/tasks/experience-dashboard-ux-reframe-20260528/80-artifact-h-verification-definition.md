# 산출물 H 상세 정의: 검증 기준과 완료 정의

## H-0. 이 산출물의 목적

이번 UX 재구성 작업이 `대충 문구만 바뀐 상태`로 끝나지 않도록, 코드/화면/데이터/리스크 기준의 완료 정의를 명확히 한다.

완료 보고는 단순히 `수정했습니다`가 아니라, 아래 질문에 답할 수 있어야 한다.

```text
목표 산출물 A~G가 실제로 화면에 반영되었는가?
기존 기능이 깨지지 않았는가?
데이터가 실제 기록 기반으로 보이는가?
모바일에서 깨지지 않는가?
금지 범위를 건드리지 않았는가?
다음 사람이 이어서 확인할 수 있는 handoff가 남았는가?
```

---

## H-1. 완료의 최상위 정의

이번 작업은 아래 조건을 모두 만족해야 완료로 본다.

1. 산출물 A~G의 체크리스트가 충족된다.
2. `HomeDashboard`의 첫 인상이 업무 관리 화면이 아니라 경험 정리 대시보드로 바뀐다.
3. 상단/캘린더/하단이 `기록 → 해석 → 직무 연결 → 이력서 후보 → 다음 행동` 흐름으로 읽힌다.
4. DB schema, RLS, env, deploy, main merge를 건드리지 않는다.
5. `git diff --check`가 통과한다.
6. `npm run build`가 통과하거나, 실패 시 기존 원인/환경 원인을 명확히 기록한다.
7. 최종 handoff가 남는다.

## H-2. 산출물별 완료 체크

### 산출물 A: 상단 카드

```text
[ ] 기존 상단 4개 박스가 단일 `최근 경험 분석 결과` 카드로 통합되었다.
[ ] 반복 경험 신호가 보인다.
[ ] 연결 가능성이 보이는 직무가 보인다.
[ ] 기록 이어가기 / 이력서 후보 보기 CTA가 있다.
[ ] 기록 없음 상태가 부정적으로 보이지 않는다.
```

### 산출물 B: 캘린더

```text
[ ] 캘린더가 `경험 흐름 캘린더`로 보인다.
[ ] 선택 날짜 상세가 경험 흐름 / 분석 신호 / 다음 행동 구조다.
[ ] 기록 없는 날짜에도 첫 기록 유도 CTA가 있다.
[ ] 기록 있음/오늘/선택 날짜 상태가 명확하다.
```

### 산출물 C: 하단 흐름

```text
[ ] 하단이 최근 경험 흐름 / 현재 강점 및 부족 신호 / 다음 추천 행동 3단 흐름이다.
[ ] 최근 리포트 표현이 메인 섹션명에서 사라졌다.
[ ] 추천 행동에는 이유와 기대 효과가 있다.
```

### 산출물 D: 카피/톤

```text
[ ] 업무관리 SaaS 톤보다 경험 정리/커리어 분석 톤이 우선한다.
[ ] 확정형/보장형 표현이 없다.
[ ] 부족 신호가 평가가 아니라 보완 방향으로 표현된다.
```

### 산출물 E: 데이터 연결

```text
[ ] 실제 기록이 있으면 기록 기반 신호가 우선 표시된다.
[ ] 데이터가 부족하면 fallback이 예시/시작 안내로 표시된다.
[ ] 가짜 분석처럼 보이는 하드코딩이 없다.
```

### 산출물 F: CTA 연결

```text
[ ] CTA가 기존 handler/화면 흐름과 연결된다.
[ ] AI 자동 호출이 없다.
[ ] 기능이 없는 CTA를 새로 만들지 않았다.
```

### 산출물 G: 반응형

```text
[ ] 모바일에서 가로 스크롤이 없다.
[ ] 카드/캘린더/CTA가 모바일에서 자연스럽게 1열로 쌓인다.
[ ] 데스크톱에서는 대시보드형 grid가 유지된다.
```

## H-3. 필수 코드 검증

아래 명령은 반드시 실행한다.

```powershell
git diff --check
npm run build
```

가능하면 아래도 실행한다.

```powershell
npm run lint
```

단, lint script가 없거나 기존 unrelated lint error가 있으면 실패를 이유로 전체 작업을 중단하지 않는다. 실패 원인과 영향 범위를 handoff에 기록한다.

## H-4. 필수 Git 검증

작업 종료 전 아래를 확인한다.

```powershell
git status --short
git diff --stat
git diff --check
git log --oneline -5
```

확인 기준:

- main에서 직접 작업하지 않았는가?
- unrelated 변경이 섞이지 않았는가?
- 허용 파일 외 변경이 생기지 않았는가?
- 문서/코드 변경이 의도한 범위 안에 있는가?

## H-5. 화면 QA 시나리오

가능하면 Playwright 또는 브라우저로 아래 시나리오를 확인한다.

### 시나리오 1. 기록 없는 상태

확인:

```text
상단 카드가 첫 기록 유도 상태로 보이는가?
캘린더가 빈 상태를 부정적으로 반복하지 않는가?
하단이 첫 기록을 유도하는가?
```

### 시나리오 2. 기록 1~2개 상태

확인:

```text
초기 분석으로 조심스럽게 표현되는가?
경험 신호가 1~3개 보이는가?
더 기록하면 좋아진다는 안내가 있는가?
```

### 시나리오 3. 기록 여러 개 상태

확인:

```text
반복 경험 신호가 보이는가?
직무 연결 가능성이 보이는가?
이력서 후보 CTA가 자연스러운가?
```

### 시나리오 4. 캘린더 날짜 선택

확인:

```text
기록 있는 날짜 선택 시 경험 흐름/신호/다음 행동이 보이는가?
기록 없는 날짜 선택 시 첫 기록 CTA가 보이는가?
```

### 시나리오 5. 모바일 화면

확인 폭:

```text
375px
430px
768px
```

확인:

```text
가로 스크롤이 없는가?
CTA가 카드 밖으로 넘치지 않는가?
캘린더 날짜 셀이 깨지지 않는가?
```

## H-6. 금지 범위 확인

작업 완료 전 아래 금지 범위를 건드리지 않았는지 확인한다.

```text
Supabase schema 변경
RLS 변경
migration 추가
Vercel env 변경
새 API 추가
production deploy
main merge
force push
reset --hard
rm -rf
인증/결제/DB 구조 변경
```

건드린 흔적이 있으면 작업 완료로 보고하지 말고 즉시 리스크로 기록한다.

## H-9. 완료로 보고하면 안 되는 경우

아래 중 하나라도 해당하면 완료로 보고하지 않는다.

```text
상단 4개 박스가 그대로 남아 있음
캘린더가 기록 있음/없음 달력에 머무름
하단이 최근 기록/최근 리포트/추천 액션 병렬 구조 그대로임
최근 리포트 표현이 메인 섹션명에 남아 있음
기록 없음/0건/없습니다가 여러 곳에서 반복됨
CTA가 버튼만 있고 이유가 없음
실제 데이터가 있는데 fallback만 표시됨
모바일에서 가로 스크롤이 생김
DB/schema/env/deploy를 건드림
검증 결과를 남기지 않음
```

## H-10. 작업 중 판단 규칙

1. 체크리스트 기준으로 완료 여부를 판단한다.
2. 빌드 실패를 숨기지 않는다.
3. 화면 QA를 못 했으면 못 했다고 적는다.
4. 금지 범위를 건드려야 할 것 같으면 그 항목만 보류한다.
5. 완료 보고는 사실 기준으로 쓴다.
6. 산출물 H 체크리스트를 만족하지 못하면 완료로 보고하지 않는다.
