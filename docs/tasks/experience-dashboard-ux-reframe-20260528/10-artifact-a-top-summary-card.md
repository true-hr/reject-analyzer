# 산출물 A 상세 정의: 상단 4개 요약 박스 → 단일 `최근 경험 분석 결과` 카드

## A-0. 이 산출물의 목적

`HomeDashboard` 상단의 기존 4개 요약 박스를 제거하거나 시각적 우선순위를 낮추고, 사용자가 화면에 들어오자마자 아래 메시지를 직관적으로 느끼게 만든다.

> 내가 기록한 경험이 단순 저장되는 것이 아니라, AI가 해석해서 직무 연결 가능성과 이력서 후보로 바뀌고 있구나.

이 산출물은 이번 UX 개편의 첫인상을 결정하는 핵심 영역이다. 단순 문구 수정이 아니라, 상단 정보 구조 자체를 `경험 분석 카드` 중심으로 재구성해야 한다.

---

## A-1. 현재 제거/대체해야 할 대상

현재 상단 4개 요약 박스는 아래 의미를 가진다.

```text
이번 주 기록 상태
이번 주 해낸 일
이번 주 만든 이력서 문장
쌓이고 있는 커리어 방향
```

이 4개 박스는 아래 이유로 그대로 유지하면 안 된다.

1. `이번 주 기록 상태`는 기록 횟수/상태를 강조해 사용자가 부족하다는 느낌을 받을 수 있다.
2. `이번 주 해낸 일`은 경험의 가치보다 업무 기록 목록처럼 보일 수 있다.
3. `이번 주 만든 이력서 문장`은 PASSMAP이 단순히 이력서 문장을 만들어주는 서비스처럼 보이게 한다.
4. `쌓이고 있는 커리어 방향`은 추상적이고 사용자가 바로 행동하기 어렵다.

따라서 상단 4개 박스를 그대로 `map()`으로 렌더링하는 구조는 산출물 A 완료 기준에 맞지 않는다.

## A-2. 목표 UI 이름

상단 단일 카드의 이름은 아래를 기본값으로 한다.

```text
최근 경험 분석 결과
```

문맥상 더 자연스러운 경우 아래 후보 중 하나를 사용할 수 있다.

```text
내 경험 분석 결과
최근 경험 연결 신호
이번 주 경험 분석 요약
```

우선순위는 다음과 같다.

1. `최근 경험 분석 결과`
2. `내 경험 분석 결과`
3. `최근 경험 연결 신호`

## A-3. 상단 카드의 시각적 역할

이 카드는 화면 최상단에서 가장 먼저 보이는 hero summary card여야 한다.

디자인 방향:

- 기존 4개 작은 박스보다 더 넓고 명확한 단일 카드
- 카드 하나 안에서 `분석 결과 → 연결 직무 → 다음 행동`이 한 번에 읽히는 구조
- 지나치게 리포트/관리자 화면처럼 딱딱하지 않게 구성
- 너무 마케팅 랜딩페이지처럼 과장하지 않음
- PASSMAP의 제품 정체성인 `기록 기반 커리어 분석`이 드러나야 함

권장 레이아웃:

```text
┌────────────────────────────────────────────────────────────┐
│ 최근 경험 분석 결과                              상태 배지 │
│ 당신의 기록에서 반복적으로 나타나는 경험 신호를 바탕으로   │
│ 연결 가능한 직무와 다음 행동을 정리했어요.                 │
│                                                            │
│ [반복 경험 신호]                                           │
│  [문제 해결 및 후속 실행] [협업 커뮤니케이션] [운영 정리]  │
│                                                            │
│ [연결 가능성이 보이는 직무]                                │
│  PM / 운영기획 / 서비스기획                                │
│                                                            │
│ [기록 이어가기] [이력서 후보 보기]                         │
└────────────────────────────────────────────────────────────┘
```

## A-4. 상세 디자인 기준

### A-4-1. 카드 외형

상단 카드는 아래 디자인 톤을 따른다.

- `rounded-2xl` 또는 `rounded-[24px]` 수준의 둥근 카드
- `border border-violet-100` 또는 기존 프로젝트의 primary/violet 계열 border
- 배경은 `white` 또는 아주 옅은 violet/slate gradient
- shadow는 과하지 않게 `shadow-sm` 또는 기존 HomeDashboard 톤과 맞춤
- 내부 padding은 데스크톱 기준 `p-5` 이상, 가능하면 `p-6`
- 모바일에서는 1열, 데스크톱에서는 좌우 2열 구조 가능

예시 Tailwind 방향:

```text
rounded-2xl border border-violet-100 bg-white p-5 shadow-sm
```

또는:

```text
rounded-[24px] border border-violet-100 bg-gradient-to-br from-white to-violet-50/40 p-6 shadow-sm
```

### A-4-2. 카드 내부 레이아웃

카드 내부는 아래 4개 블록으로 나뉜다.

1. 헤더 영역
2. 반복 경험 신호 영역
3. 연결 가능 직무 영역
4. CTA 영역

데스크톱 권장 구조:

```text
좌측 70%:
- 제목
- 설명
- 반복 경험 신호

우측 30%:
- 연결 직무
- 분석 상태/요약
- CTA 버튼
```

모바일 권장 순서:

```text
제목
설명
반복 경험 신호
연결 직무
CTA 버튼
```

## A-5. 헤더 영역 상세

### 제목

기본 제목:

```text
최근 경험 분석 결과
```

### 보조 설명

기본 설명:

```text
당신의 기록에서 반복적으로 나타나는 경험 신호를 바탕으로, 연결 가능한 직무와 다음 행동을 정리했어요.
```

기록이 없는 경우:

```text
아직 기록이 없어도 괜찮아요. 오늘 한 일을 한 줄만 남기면 AI가 경험 신호를 찾아드릴게요.
```

기록이 적은 경우:

```text
아직 기록은 많지 않지만, 남겨둔 경험에서 보이는 초기 신호를 정리했어요.
```

기록이 충분한 경우:

```text
최근 기록에서 반복적으로 나타나는 경험 신호와 직무 연결 가능성을 정리했어요.
```

### 상태 배지

카드 우측 상단 또는 제목 옆에는 작은 상태 배지를 둘 수 있다.

기록이 충분한 경우:

```text
기록 기반 분석
```

기록이 적은 경우:

```text
초기 분석
```

기록이 없는 경우:

```text
첫 기록 대기 중
```

주의: `0건`, `없음`, `미완료` 같은 부정적 표현을 상태 배지에 쓰지 않는다.

## A-6. 반복 경험 신호 영역 상세

### 영역 제목

```text
반복적으로 보이는 경험 신호
```

또는 더 짧게:

```text
반복 경험 신호
```

### 표시 방식

경험 신호는 bullet list보다 chip/card 형태를 우선한다.

권장 UI:

```text
[문제 해결 및 후속 실행] [협업 커뮤니케이션] [운영 프로세스 정리]
```

각 chip은 아래 스타일을 권장한다.

- `rounded-full`
- `border`
- `bg-violet-50`, `bg-slate-50`, `bg-emerald-50` 등 과하지 않은 배경
- text는 `text-xs` 또는 `text-sm`
- 너무 많은 색상을 쓰지 않는다.

### 표시 개수

- 기본 3개 표시
- 실제 데이터가 많아도 최대 4개까지만 표시
- 5개 이상이면 UX가 복잡해지므로 잘라낸다.

### 데이터 생성 우선순위

경험 신호는 아래 우선순위로 만든다.

1. 최근 `work_records`의 `strength_tags`, `skill_tags`, `raw_payload` 기반
2. 기존 helper `buildExperienceSignalsFromRecord` 사용 가능 시 활용
3. 최근 기록 title/description에서 기존 코드가 이미 추출하는 tag/signal 활용
4. `homeDashboardMock.js`의 demo/fallback 데이터
5. fallback 고정값

fallback 고정값:

```text
문제 해결 및 후속 실행
협업 커뮤니케이션
운영 프로세스 정리
```

### 표현 원칙

확정형보다 가능성형을 쓴다.

권장:

```text
반복적으로 나타나고 있어요
강하게 보입니다
연결 가능성이 있습니다
```

피해야 함:

```text
확실합니다
검증되었습니다
이 직무가 맞습니다
합격 가능성이 높습니다
```

## A-7. 연결 가능 직무 영역 상세

### 영역 제목

기본값:

```text
연결 가능성이 보이는 직무
```

대체 가능:

```text
현재 연결 가능한 방향
```

### 직무 표시

직무는 텍스트 또는 chip으로 표시한다.

권장 예시:

```text
PM / 운영기획 / 서비스기획
```

또는 chip:

```text
[PM] [운영기획] [서비스기획]
```

### 데이터 우선순위

직무는 아래 우선순위로 표시한다.

1. `currentCareerRoleLabel`이 있으면 최우선 사용
2. 기존 HomeDashboard 또는 App 상태에서 목표 직무 label이 있으면 사용
3. 최근 기록/경험 신호에서 추론 가능한 직무가 있으면 사용
4. 없으면 fallback

fallback:

```text
PM / 운영기획 / 서비스기획
```

### 설명 문구

직무 영역 아래에는 한 줄 설명을 둔다.

기본:

```text
최근 기록의 문제 해결·협업·운영 정리 신호가 위 직무와 연결될 수 있어요.
```

데이터가 부족한 경우:

```text
기록을 더 남기면 연결 직무를 더 정확하게 좁혀볼 수 있어요.
```

## A-8. CTA 영역 상세

### CTA 1: 기록 이어가기

기본 문구:

```text
기록 이어가기
```

역할:

- 사용자가 경험 기록/업무 기록 입력으로 이동하게 한다.
- 기존 handler를 재사용한다.
- 새 route를 만들지 않는다.

연결 후보:

```text
onOpenUpdateView
onOpenAnalysis
기존 기록 추가 handler
기존 jobSidebarView 전환
```

정확한 연결은 현재 코드에서 가장 안전한 기존 흐름을 사용한다.

### CTA 2: 이력서 후보 보기

기본 문구:

```text
이력서 후보 보기
```

역할:

- 사용자가 기록 기반 이력서 후보/초안/결과 화면으로 이동하게 한다.
- AI를 화면 진입 시 자동 호출하지 않는다.
- 버튼 클릭 시 기존 이력서 보기 또는 후보 생성 흐름으로 연결한다.

연결 후보:

```text
onOpenResumeView
setCurrentScreen("result")
handleAiResumeGenerate
handleAiResumeGenerateWithSave
```

주의:
저장되지 않은 입력이 있는 경우 무리하게 이력서 생성하지 않는다. 기존 저장 후 생성 흐름이 있으면 사용하고, 없으면 기존 이력서 보기 화면으로 이동한다.

### CTA 버튼 디자인

CTA는 2개를 나란히 둔다.

1차 CTA:

```text
기록 이어가기
```

- primary/violet filled button
- 사용자의 다음 행동을 시작시키는 버튼

2차 CTA:

```text
이력서 후보 보기
```

- outline 또는 white button
- 기존 이력서 후보/결과 확인

모바일에서는 버튼이 1열로 떨어져도 자연스러워야 한다.

## A-9. 기록 상태별 UI 분기

상단 카드는 기록 상태에 따라 다르게 보여야 한다.

### Case 1. 기록이 없는 사용자

목표:
- 빈 상태를 부정적으로 보이지 않게 한다.
- `첫 기록을 남기면 분석이 시작된다`는 기대를 준다.

카피:

```text
최근 경험 분석 결과
아직 기록이 없어도 괜찮아요. 오늘 한 일을 한 줄만 남기면 AI가 경험 신호를 찾아드릴게요.
```

경험 신호 영역:

```text
첫 기록을 남기면 이런 신호를 찾아볼 수 있어요
[문제 해결] [협업 경험] [성과 정리]
```

직무 영역:

```text
기록이 쌓이면 연결 가능한 직무를 함께 정리해드릴게요.
```

CTA:

```text
첫 경험 기록하기
```

2차 CTA는 숨기거나 약하게 표시한다. 예시 보기 기능이 없으면 새로 만들지 않는다.

### Case 2. 기록이 1~2개인 사용자

목표:
- 너무 확정적으로 분석하지 않는다.
- 초기 신호를 보여주되 더 기록하면 좋아진다고 안내한다.

카피:

```text
아직 기록은 많지 않지만, 남겨둔 경험에서 보이는 초기 신호를 정리했어요.
```

상태 배지:

```text
초기 분석
```

설명:

```text
기록을 몇 개 더 남기면 반복 경험과 연결 직무를 더 정확하게 볼 수 있어요.
```

CTA:

```text
기록 이어가기
이력서 후보 보기
```

### Case 3. 기록이 충분한 사용자

목표:
- 기록 기반 분석이 진행되고 있다는 느낌을 준다.
- 경험 신호와 직무 연결을 명확히 보여준다.

카피:

```text
최근 기록에서 반복적으로 나타나는 경험 신호와 직무 연결 가능성을 정리했어요.
```

상태 배지:

```text
기록 기반 분석
```

경험 신호:
- 실제 기록 기반 3~4개
- fallback 최소화

직무:
- currentCareerRoleLabel 우선
- 없으면 추론/fallback

CTA:

```text
기록 이어가기
이력서 후보 보기
```

## A-10. 기존 4개 요약 정보의 흡수 방식

기존 4개 박스의 의미를 완전히 버리지 말고, 카드 내부에 자연스럽게 흡수한다.

| 기존 박스 | 새 카드에서의 역할 |
|---|---|
| 이번 주 기록 상태 | 상태 배지 또는 작은 보조 문구 |
| 이번 주 해낸 일 | 반복 경험 신호 / 최근 경험 요약 |
| 이번 주 만든 이력서 문장 | 이력서 후보 보기 CTA |
| 쌓이고 있는 커리어 방향 | 연결 가능성이 보이는 직무 |

즉, 기존 정보를 없애는 것이 아니라 `기록 횟수 중심`에서 `경험 해석 중심`으로 재배치한다.

## A-11. 피해야 할 UI/카피

아래처럼 구현되면 산출물 A는 미완료로 본다.

### 피해야 할 구조

```text
[이번 주 기록 상태] [이번 주 해낸 일] [이번 주 만든 이력서 문장] [쌓이고 있는 커리어 방향]
```

기존 4박스가 그대로 남아 있으면 안 된다.

### 피해야 할 카피

```text
아직 기록이 없습니다.
0건
이력서 문장을 만들었습니다.
쌓이고 있는 커리어 방향
업무 관리 상태
업데이트 필요
```

### 피해야 할 제품 인상

- 생산성 앱
- 업무 체크리스트
- 단순 기록 저장소
- 이력서 문장 생성기
- 관리자 리포트 화면

### 지향해야 할 제품 인상

- 경험 정리 대시보드
- 커리어 분석 플랫폼
- 기록 기반 직무 연결
- 이력서 후보 발굴
- 다음 행동을 알려주는 취업/이직 준비 도구

## A-12. 접근성/반응형 기준

### 접근성

- CTA는 `button`이어야 한다.
- 클릭 가능한 요소는 hover/focus 스타일이 있어야 한다.
- 상태 배지나 chip만으로 중요한 의미를 전달하지 말고, 텍스트 설명을 함께 둔다.
- 색상만으로 상태를 구분하지 않는다.

### 반응형

모바일:

- 카드 내부는 1열
- chip은 줄바꿈
- CTA는 세로 배치 가능
- 텍스트가 카드 밖으로 넘치면 안 됨

데스크톱:

- 카드 내부는 2열 가능
- 좌측: 제목/설명/경험 신호
- 우측: 연결 직무/CTA

산출물 A는 아래 항목이 모두 충족되어야 완료로 본다.

[ ] 상단 카드 제목이 “최근 경험 분석 결과” 또는 동등한 표현으로 바뀌었다.
[ ] 카드 안에 반복 경험 신호 3개 내외가 표시된다.
[ ] 카드 안에 연결 가능성이 보이는 직무가 표시된다.
[ ] 기록이 있는 경우 실제 기록 기반 신호를 우선 사용한다.
[ ] 기록이 없는 경우 부정적 빈 상태 대신 첫 기록 유도 문구를 보여준다.
[ ] CTA는 “기록 이어가기”와 “이력서 후보 보기” 중심으로 구성된다.
[ ] CTA에는 왜 눌러야 하는지 이해되는 문맥이 있다.
[ ] “이력서 문장 생성 서비스”처럼 보이지 않도록 “이력서 후보” 표현을 우선 사용한다.

[ ] PR #574 충돌 가능성이 높은 파일은 건드리지 않았다. 건드렸다면 이유를 최종 보고에 적었다.

## A-15. 산출물 A 검증 방법

```powershell

```

### 화면 검증

## A-16. 산출물 A 작업 중 판단 규칙

작업 중 애매하면 아래 기준을 따른다.

1. 실제 기록 기반 표시를 우선한다.
2. 기록이 없으면 부담 낮은 첫 기록 유도 문구를 쓴다.
3. 확정형보다 가능성형 표현을 쓴다.
4. `이력서 문장`보다 `이력서 후보` 표현을 우선한다.
5. 기존 4개 박스의 정보를 버리지 말고 단일 카드 안에 흡수한다.
6. 디자인은 기존 PASSMAP violet/slate 톤을 따른다.
7. DB/API/env 변경이 필요하면 하지 않는다.
8. PR #574 충돌 가능성이 큰 파일은 피한다.
9. 사용자에게 질문하지 않고 이 문서의 기본값으로 진행한다.
10. 산출물 A 완료 체크리스트를 만족하지 못하면 완료로 보고하지 않는다.
