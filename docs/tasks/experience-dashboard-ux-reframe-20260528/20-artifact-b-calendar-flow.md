# 산출물 B 상세 정의: 캘린더 → `경험 흐름 캘린더`

## B-0. 이 산출물의 목적

`HomeDashboard`의 캘린더 영역을 단순히 날짜별 기록 여부를 확인하는 달력이 아니라, 사용자가 자신의 경험이 어떻게 쌓이고 해석되는지 확인하는 `경험 흐름 캘린더`로 재구성한다.

사용자가 캘린더를 봤을 때 아래 메시지를 느껴야 한다.

> 기록한 날짜마다 경험 신호가 쌓이고, 그 신호가 직무 연결과 이력서 후보로 이어지고 있구나.

이 산출물은 `기록 빈도 확인`보다 `경험 축적 흐름 확인`을 우선한다.

---

## B-1. 현재 제거/대체해야 할 인상

현재 캘린더는 아래 인상에 가깝다.

```text
업무를 기록한 날짜 확인
기록이 있는 날짜와 없는 날짜 구분
선택 날짜의 기록 리스트 확인
```

이번 산출물 B에서는 아래 인상이 되면 미완료다.

- 생산성 앱의 출석 체크 달력
- 업무 일지 보관함
- 기록 있음/없음만 보여주는 달력
- `기록 없음`, `0건`, `없습니다`가 반복되는 빈 상태 화면

목표 인상은 아래다.

- 경험 흐름 대시보드
- 기록 기반 커리어 신호 지도
- 어떤 경험이 쌓였는지 확인하는 캘린더
- 다음에 무엇을 보완해야 하는지 알려주는 캘린더

---

## B-2. 목표 UI 이름

캘린더 섹션의 기본 제목은 아래를 사용한다.

```text
경험 흐름 캘린더
```

대체 후보:

```text
내 경험 흐름
이번 달 경험 흐름
경험 축적 캘린더
```

우선순위는 다음과 같다.

1. `경험 흐름 캘린더`
2. `이번 달 경험 흐름`
3. `경험 축적 캘린더`

---

## B-3. 캘린더 섹션의 전체 레이아웃

캘린더 섹션은 아래 3개 영역으로 구성한다.

1. 섹션 헤더
2. 월간 캘린더 본문
3. 선택 날짜 상세/추천 행동 패널

권장 레이아웃:

```text
┌────────────────────────────────────────────────────────────┐
│ 경험 흐름 캘린더                                           │
│ 기록한 날짜마다 어떤 경험 신호가 쌓였는지 확인해보세요.    │
│                                                            │
│ ┌──────────── 월간 캘린더 ────────────┐ ┌ 선택 날짜 상세 ┐ │
│ │ 날짜 셀                             │ │ 이 날짜의 흐름 │ │
│ │ - 기록 있음 dot                     │ │ 분석 신호      │ │
│ │ - 경험 신호 색/칩                   │ │ 다음 행동      │ │
│ └─────────────────────────────────────┘ └────────────────┘ │
└────────────────────────────────────────────────────────────┘
```

데스크톱:

- 좌측: 캘린더
- 우측: 선택 날짜 상세 패널
- 가능하면 `lg:grid-cols-[minmax(0,1.3fr)_minmax(280px,0.7fr)]` 같은 비율

모바일:

- 상단: 캘린더
- 하단: 선택 날짜 상세 패널
- 가로 스크롤 없이 1열 배치

## B-4. 섹션 헤더 상세

### 제목

```text
경험 흐름 캘린더
```

### 설명 문구

기본 설명:

```text
기록한 날짜마다 어떤 경험 신호가 쌓였는지 확인하고, 부족한 부분을 다음 기록으로 보완해보세요.
```

기록이 없는 경우:

```text
아직 표시할 기록은 없지만, 오늘 한 일을 한 줄만 남기면 경험 흐름이 이곳에 쌓이기 시작해요.
```

기록이 적은 경우:

```text
아직 기록은 많지 않지만, 남겨둔 날짜의 경험 신호를 먼저 정리했어요.
```

기록이 충분한 경우:

```text
이번 달 기록에서 반복되는 경험 신호와 보완하면 좋은 지점을 함께 볼 수 있어요.
```

### 보조 상태 배지

캘린더 헤더 우측에는 작은 상태 배지를 둘 수 있다.

- 기록 없음: `첫 기록 대기 중`
- 기록 1~2개: `초기 흐름`
- 기록 3개 이상: `경험 흐름 분석 중`

주의:

`0건`, `기록 없음`, `비어 있음` 같은 부정적 표현을 배지로 쓰지 않는다.

## B-5. 월간 캘린더 본문 상세

### 날짜 셀의 기본 구조

각 날짜 셀에는 아래 요소를 둘 수 있다.

1. 날짜 숫자
2. 기록 존재 표시 dot 또는 작은 bar
3. 감지된 대표 경험 신호 1개
4. 선택 상태
5. 오늘 표시

권장 날짜 셀 구조:

```text
┌──────┐
│  14  │
│  ●   │
│ 협업 │
└──────┘
```

또는 공간이 좁으면:

```text
14  ●
```

### 날짜 셀 상태

아래 상태를 구분한다.

| 상태 | 의미 | UI |
|---|---|---|
| 기록 없음 | 아직 기록 없음 | 빈 셀 또는 옅은 hover만 |
| 기록 있음 | work_records 존재 | 작은 dot 또는 subtle highlight |
| 강한 신호 있음 | strength/signal 존재 | chip 또는 더 선명한 dot |
| 선택 날짜 | 사용자가 선택 | border/primary ring |
| 오늘 | today | 작은 `오늘` 표시 또는 subtle badge |

### 색상 원칙

- 너무 많은 색상을 쓰지 않는다.
- violet/slate/emerald/amber 정도만 사용한다.
- 색상만으로 의미를 전달하지 않는다.
- 신호 이름이나 tooltip/title 텍스트로 의미를 보완한다.

### 날짜 셀에서 피해야 할 것

- 날짜마다 `0건` 표시
- 기록 없는 날짜마다 `없음` 표시
- 너무 많은 텍스트
- 각 날짜 셀이 카드처럼 과하게 커지는 구조
- 모바일에서 캘린더가 가로로 넘치는 구조

---

## B-6. 선택 날짜 상세 패널

선택 날짜 상세 패널은 캘린더 오른쪽 또는 아래에 배치한다.

패널 제목 기본값:

```text
이 날짜의 경험 흐름
```

대체 후보:

```text
선택 날짜 인사이트
이 날짜의 경험 신호
```

### 기록이 있는 날짜의 상세 구조

기록이 있는 날짜는 아래 순서로 보여준다.

1. 날짜와 간단 요약
2. 기록 요약
3. 감지된 경험 신호
4. 연결 가능한 역량/직무
5. 다음 보완 행동

권장 UI:

```text
이 날짜의 경험 흐름

오늘 남긴 기록에서 아래 신호가 보입니다.

기록 요약
- 고객 문의 흐름을 정리하고 대응 기준을 문서화함

감지된 경험 신호
[문제 해결] [협업 커뮤니케이션] [운영 정리]

다음 보완 행동
성과 수치나 사용자 반응을 추가하면 이력서 후보의 설득력이 높아져요.

[이 날짜 기록 보완하기]
```

### 기록이 없는 날짜의 상세 구조

기록이 없는 날짜는 `없습니다`를 반복하지 않고, 바로 기록을 시작할 수 있게 안내한다.

권장 UI:

```text
이 날짜는 아직 비어 있어요

괜찮아요. 오늘 해결한 문제, 협업한 사람, 결과로 달라진 점 중 하나만 적어도 경험 신호를 찾을 수 있어요.

예시
- 오늘 처리한 이슈
- 협업한 사람/부서
- 결과로 달라진 점

[이 날짜에 경험 남기기]
```

주의:

`선택한 날짜에 기록된 업무가 아직 없습니다` 문장을 단독으로 노출하지 않는다. 쓸 경우에도 CTA/예시와 함께 묶는다.

---

## B-7. 월간 요약 영역

캘린더 주변에는 이번 달/이번 주의 흐름 요약을 작게 보여줄 수 있다.

권장 제목:

```text
이번 달 경험 흐름 요약
```

표시 항목:

- 기록이 집중된 주
- 반복적으로 보이는 경험 신호
- 아직 부족한 신호
- 다음에 보완하면 좋은 정보

권장 예시:

```text
이번 달에는 협업 커뮤니케이션과 문제 해결 신호가 반복적으로 보이고 있어요.
다음 기록에서는 성과 수치나 사용자 반응을 함께 남기면 이력서 후보가 더 선명해집니다.
```

기록이 없는 경우:

```text
첫 기록을 남기면 이번 달 경험 흐름 요약이 자동으로 쌓여요.
```

## B-8. 추천 행동 패널

선택 날짜 상세 패널 하단 또는 캘린더 섹션 하단에는 `다음 추천 행동`을 둔다.

### 기록이 있는 날짜

추천 행동 예시:

```text
성과 정보 보완하기
이 기록에는 실행 과정은 보이지만 결과 수치가 부족합니다. 처리 건수, 개선률, 사용자 반응을 추가하면 이력서 후보의 설득력이 높아져요.
```

CTA:

```text
기록 보완하기
```

### 기록이 없는 날짜

추천 행동 예시:

```text
한 줄로 시작하기
오늘 해결한 문제나 협업한 내용을 한 줄만 남겨도 경험 흐름이 쌓이기 시작합니다.
```

CTA:

```text
이 날짜에 경험 남기기
```

### 기록이 충분한 날짜

추천 행동 예시:

```text
이력서 후보로 확인하기
이 날짜의 기록은 문제 해결과 협업 조율 신호가 함께 보여요. 이력서 후보에서 어떻게 정리되는지 확인해보세요.
```

CTA:

```text
이력서 후보 보기
```

## B-9. 데이터 생성 우선순위

캘린더의 경험 흐름 데이터는 아래 우선순위로 만든다.

1. 실제 `work_records` 날짜별 기록
2. 기록의 `strength_tags`, `skill_tags`, `raw_payload`, description/title
3. 기존 `homeDashboardCalendarUtils.js` view model
4. 기존 `homeDashboardMock.js` demo data
5. fallback 문구

DB schema 변경 없이 구현한다.

## B-10. 기존 정보의 흡수 방식

기존 캘린더/날짜 상세의 의미를 버리지 말고 아래처럼 흡수한다.

| 기존 요소 | 새 구조에서의 역할 |
|---|---|
| 날짜별 기록 여부 | 경험 흐름 dot/highlight |
| 선택 날짜 기록 목록 | 이 날짜의 경험 흐름 |
| 날짜별 강점 | 감지된 경험 신호 |
| 월간 자산 요약 | 이번 달 경험 흐름 요약 |
| 이 날짜에 기록 추가 | 이 날짜에 경험 남기기 CTA |

## B-11. 피해야 할 UI/카피

아래 표현이 반복되면 산출물 B는 미완료다.

```text
기록 없음
0건
없습니다
표시할 강점이 없습니다
선택한 날짜에 기록된 업무가 아직 없습니다
```

완전히 금지하지는 않지만, 단독/반복 노출하지 않는다. 필요하면 다음 행동 CTA와 함께 묶는다.

피해야 할 제품 인상:

- 출석 체크
- 업무 일지 달력
- 기록 건수 집계표
- 빈 상태를 계속 지적하는 화면

지향해야 할 제품 인상:

- 경험 흐름 캘린더
- 커리어 신호 지도
- 다음 기록 방향을 알려주는 도구

## B-12. 접근성/반응형 기준

### 접근성

- 날짜 셀은 키보드 focus가 가능해야 한다.
- 선택 상태는 색상만으로 구분하지 않는다.
- 날짜 버튼에는 aria-label 또는 title로 날짜/기록 상태 의미를 전달한다.
- CTA는 button으로 구현한다.

### 반응형

모바일:

- 캘린더와 상세 패널은 1열로 쌓인다.
- 날짜 셀 텍스트가 넘치지 않는다.
- CTA는 세로 배치 가능하다.

데스크톱:

- 캘린더와 상세 패널은 2열 구조가 가능하다.
- 상세 패널은 캘린더보다 시각적 우선순위가 낮아도 되지만, 선택 날짜의 의미는 명확해야 한다.

## B-14. 산출물 B 완료 체크리스트

```text
[ ] 캘린더 섹션 제목이 `경험 흐름 캘린더` 또는 동등한 표현으로 바뀌었다.
[ ] 캘린더가 단순 기록 있음/없음 달력처럼 보이지 않는다.
[ ] 선택 날짜 상세가 `경험 흐름 / 분석 신호 / 다음 행동` 구조로 읽힌다.
[ ] 기록 없는 날짜도 부정적 빈 상태가 아니라 첫 기록 CTA로 이어진다.
[ ] `기록 없음`, `0건`, `없습니다`류 표현이 캘린더 영역에서 반복 강조되지 않는다.
[ ] 기록이 있는 날짜에는 실제 기록 기반 신호가 우선 표시된다.
[ ] 이번 달 또는 이번 주 경험 흐름 요약이 있다.
[ ] 날짜별 선택/오늘/기록 있음 상태가 시각적으로 구분된다.
[ ] 모바일에서 캘린더와 상세 패널이 자연스럽게 1열로 쌓인다.
[ ] DB schema / RLS / env / API 변경 없이 구현되었다.
```

## B-15. 산출물 B 검증 방법

### 코드 검증

```powershell
git diff --check
npm run build
```

### 화면 검증

아래 상태를 확인한다.

1. 기록 없는 사용자 또는 mock fallback
2. 기록이 1~2개인 사용자
3. 기록이 여러 날짜에 있는 사용자
4. 오늘 날짜 선택
5. 기록 없는 날짜 선택
6. 기록 있는 날짜 선택
7. 모바일 폭
