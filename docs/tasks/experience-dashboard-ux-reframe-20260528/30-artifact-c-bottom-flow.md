# 산출물 C 상세 정의: 하단 3단 흐름

## C-0. 이 산출물의 목적

`HomeDashboard` 하단 영역을 기존의 `최근 기록 / 최근 리포트 / 추천 액션` 병렬 구조에서 벗어나, 사용자가 자신의 기록이 어떻게 커리어 자산으로 해석되는지 이해할 수 있는 3단 흐름으로 재구성한다.

사용자는 하단을 보면서 아래 흐름을 자연스럽게 이해해야 한다.

> 최근에 남긴 경험이 있고, 그 경험에서 강점/부족 신호가 읽히며, 다음에는 무엇을 보완해야 하는지 알 수 있다.

이 산출물의 핵심은 `정보 나열`이 아니라 `기록 → 해석 → 행동` 흐름이다.

---

## C-1. 현재 제거/대체해야 할 구조

현재 하단은 아래 구조에 가깝다.

```text
최근 기록
최근 리포트
추천 액션
```

이 구조는 각각의 섹션이 분리되어 있어, 사용자가 아래 연결을 이해하기 어렵다.

```text
내 기록이 어떻게 해석되었는가?
이 기록이 어떤 직무/역량과 연결되는가?
다음에는 무엇을 적어야 하는가?
```

따라서 이번 산출물 C에서는 하단을 아래 3개 흐름으로 바꾼다.

```text
최근 경험 흐름
현재 강점 및 부족 신호
다음 추천 행동
```

## C-2. 목표 UI 이름과 구성

하단 전체 영역의 상위 제목은 아래 중 하나를 사용한다.

```text
최근 커리어 인사이트
```

대체 후보:

```text
내 경험 흐름 요약
최근 경험 분석
다음 커리어 행동
```

우선순위:

1. `최근 커리어 인사이트`
2. `내 경험 흐름 요약`
3. `최근 경험 분석`

하단 내부 3개 카드/섹션은 아래 이름을 기본값으로 한다.

```text
최근 경험 흐름
현재 강점 및 부족 신호
다음 추천 행동
```

## C-3. 전체 레이아웃

데스크톱에서는 3개 카드를 가로 또는 2+1 grid로 배치할 수 있다.

권장 데스크톱 레이아웃:

```text
┌────────────────────────────────────────────────────────────┐
│ 최근 커리어 인사이트                                      │
│ 기록이 어떻게 커리어 자산으로 바뀌고 있는지 확인해보세요. │
│                                                            │
│ ┌ 최근 경험 흐름 ┐ ┌ 현재 강점 및 부족 신호 ┐ ┌ 다음 행동 ┐ │
│ │ 최근 기록 요약 │ │ 강점 / 보완 신호       │ │ 추천 CTA  │ │
│ └────────────────┘ └────────────────────────┘ └───────────┘ │
└────────────────────────────────────────────────────────────┘
```

모바일에서는 아래 순서로 1열 배치한다.

```text
최근 커리어 인사이트
최근 경험 흐름
현재 강점 및 부족 신호
다음 추천 행동
```

시선 흐름은 반드시 `최근 경험 흐름 → 현재 신호 → 다음 행동` 순서여야 한다.

## C-4. 섹션 1: 최근 경험 흐름

### 목적

`최근 기록`을 단순 리스트로 보여주는 것이 아니라, 최근 기록들이 어떤 경험 흐름으로 읽히는지 보여준다.

### 제목

```text
최근 경험 흐름
```

대체 후보:

```text
최근 기록에서 보이는 흐름
최근 경험 요약
```

### 포함 요소

이 카드에는 아래 요소가 포함되어야 한다.

1. 최근 기록 1~3개 요약
2. 각 기록에서 보이는 대표 경험 신호
3. 연결 가능한 역량/직무 힌트
4. 기록이 없는 경우 첫 기록 유도

### 기록이 있는 경우 예시

```text
최근 기록에서는 협업 조율과 문제 해결 신호가 반복적으로 보이고 있어요.

- 고객 문의 흐름 정리 → 운영 프로세스 정리
- 개발팀과 이슈 확인 → 협업 커뮤니케이션
- 후속 조치 기준 정리 → 문제 해결 및 후속 실행
```

### 기록이 없는 경우 예시

```text
아직 최근 경험 흐름을 만들 기록이 충분하지 않아요.
오늘 해결한 문제나 협업한 내용을 한 줄만 남기면 이곳에 경험 흐름이 쌓이기 시작합니다.
```

CTA:

```text
첫 경험 기록하기
```

### 디자인 기준

- 단순 table보다 card/list 구조를 우선한다.
- 각 기록은 너무 길게 보여주지 않는다.
- 기록 제목/요약은 1~2줄로 제한한다.
- 경험 신호는 chip 형태를 우선한다.

## C-5. 섹션 2: 현재 강점 및 부족 신호

### 목적

사용자가 자신의 기록에서 어떤 강점이 보이고, 어떤 정보가 아직 부족한지 이해하게 한다.

### 제목

```text
현재 강점 및 부족 신호
```

대체 후보:

```text
강하게 보이는 신호와 보완할 신호
현재 커리어 신호
```

### 카드 구조

이 카드는 아래 2단 구조를 권장한다.

```text
강하게 보이는 신호
[협업 커뮤니케이션] [문제 해결] [운영 정리]

보완하면 좋은 신호
[정량 성과] [사용자 반응] [비즈니스 임팩트]
```

### 강점 신호 예시

```text
협업 커뮤니케이션은 최근 기록에서 비교적 자주 보입니다.
여러 이해관계자와 논의하고 기준을 맞춘 경험이 강점으로 읽힐 수 있어요.
```

### 부족 신호 예시

```text
성과 수치와 결과 변화는 아직 약하게 보입니다.
처리 건수, 개선률, 사용자 반응을 추가하면 이력서 후보의 설득력이 높아져요.
```

### 데이터 우선순위

강점 신호:

1. 실제 work_records의 strength_tags
2. skill_tags
3. raw_payload/description 기반 신호
4. 기존 helper 신호
5. fallback

부족 신호:

1. 최근 기록에 없는 정보 패턴 감지
2. 기존 improvement hints
3. 직무별 추천 보완 요소
4. fallback

기본 부족 신호 fallback:

```text
정량 성과
사용자 반응
비즈니스 임팩트
```

### 주의

부족 신호는 사용자를 평가하거나 낮추는 톤으로 쓰지 않는다.

피해야 할 표현:

```text
부족합니다
약합니다
문제가 있습니다
```

권장 표현:

```text
아직 덜 드러납니다
더 선명하게 만들 수 있어요
보완하면 설득력이 높아져요
```

## C-6. 섹션 3: 다음 추천 행동

### 목적

사용자가 다음에 무엇을 해야 하는지 바로 결정할 수 있게 한다.

### 제목

```text
다음 추천 행동
```

대체 후보:

```text
다음에 하면 좋은 일
다음 기록 방향
```

### 기본 구조

추천 행동은 단순 버튼 나열이 아니라 아래 구조를 가진다.

```text
추천 행동 제목
왜 필요한지 설명
기대 효과
CTA
```

### 추천 행동 예시 1: 정량 성과 보완

```text
정량 성과 보완하기
최근 기록에는 실행 과정은 잘 남아 있지만 결과 수치가 아직 덜 드러납니다.
처리 건수, 개선률, 사용자 반응을 추가하면 PM/운영기획 직무에서 이력서 후보의 설득력이 높아져요.
[성과 정보 추가하기]
```

### 추천 행동 예시 2: 협업 맥락 보완

```text
협업 맥락 선명하게 만들기
기록에 협업 정황은 보이지만 누구와 어떤 기준을 맞췄는지가 더 드러나면 좋아요.
팀, 부서, 고객, 개발자 등 이해관계자를 함께 적으면 커뮤니케이션 역량이 더 분명해집니다.
[협업 맥락 보완하기]
```

### 추천 행동 예시 3: 이력서 후보 확인

```text
이력서 후보로 확인하기
최근 기록에서 문제 해결과 운영 정리 신호가 보입니다.
이 경험이 이력서 후보에서 어떻게 정리되는지 확인해보세요.
[이력서 후보 보기]
```

### 추천 행동 개수

- 기본 2~3개
- 너무 많으면 체크리스트처럼 보여서 피한다.
- 1개만 있을 때는 카드 하나를 크게 보여줘도 된다.

## C-7. `최근 리포트` 표현 대체 규칙

하단 메인 카피에서 `최근 리포트` 표현은 사용하지 않는다.

대체 표현:

```text
최근 커리어 인사이트
최근 경험 분석
내 경험 분석 결과
```

기본값은 아래다.

```text
최근 커리어 인사이트
```

`리포트`라는 단어가 필요한 경우에는 보조 설명에서만 제한적으로 쓴다. 메인 섹션명에는 쓰지 않는다.

## C-8. 데이터 연결 기준

1. 실제 `work_records`
2. `experience_cards`

기존 mock 데이터를 버리지 말고, 사용자 친화적인 카피로 재정리해 fallback으로 사용한다.

## C-10. 디자인 기준

- 카드 3개는 같은 visual family로 보이게 한다.
- 섹션별 역할은 명확히 구분한다.
- 배경은 white/slate/violet 계열을 사용한다.
- 강점 신호는 너무 과한 초록색으로 성공처럼 보이지 않게 한다.
- 부족 신호는 빨간색/경고처럼 보이지 않게 한다.
- CTA는 한 카드당 1개를 기본으로 한다.

권장 카드 스타일:

```text
rounded-2xl border border-slate-200 bg-white p-5 shadow-sm
```

또는 강조 카드:

```text
rounded-2xl border border-violet-100 bg-violet-50/40 p-5
```

## C-11. 접근성/반응형 기준

모바일:

- 3개 카드는 1열로 쌓인다.
- 각 카드의 제목/설명/CTA 순서가 유지된다.
- chip은 줄바꿈된다.

데스크톱:

- 3열 또는 2열+1열 가능
- 카드 높이가 지나치게 들쭉날쭉해도 기능상 문제는 없지만, 시각적 균형을 맞춘다.

접근성:

- CTA는 button으로 구현한다.
- 버튼에는 hover/focus style이 있다.
- 색상만으로 강점/보완 신호를 구분하지 않는다.

## C-13. 산출물 C 완료 체크리스트

```text
[ ] 하단이 `최근 경험 흐름 / 현재 강점 및 부족 신호 / 다음 추천 행동` 3단 흐름으로 재구성되었다.
[ ] `최근 리포트` 표현이 메인 섹션명에서 사라졌다.
[ ] 최근 기록이 단순 목록이 아니라 경험 흐름으로 해석되어 보인다.
[ ] 강점 신호와 보완 신호가 구분되어 보인다.
[ ] 부족 신호가 사용자 평가처럼 보이지 않고 보완 방향처럼 보인다.
[ ] 추천 행동에는 왜 필요한지와 기대 효과가 함께 있다.
[ ] CTA가 기존 화면/handler와 연결되어 있다.
[ ] 기록이 없는 경우 첫 기록 유도 fallback이 있다.
[ ] 모바일에서 1열로 자연스럽게 보인다.
[ ] DB schema / RLS / env / API 변경 없이 구현되었다.
```

## C-15. 작업 중 판단 규칙

1. 단순 목록보다 흐름을 우선한다.
2. 리포트보다 인사이트 표현을 우선한다.
3. 부족함보다 보완 방향을 우선한다.
4. 추천 행동에는 이유를 함께 쓴다.
5. 기존 데이터와 mock을 재사용한다.
6. DB/API/env 변경은 하지 않는다.
7. 산출물 C 완료 체크리스트를 만족하지 못하면 완료로 보고하지 않는다.
