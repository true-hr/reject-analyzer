# 산출물 F 상세 정의: CTA 연결 기준

## F-0. 이 산출물의 목적

`HomeDashboard`의 CTA가 단순 버튼 나열이 아니라, 사용자의 현재 기록 상태와 다음 행동을 자연스럽게 연결하도록 만든다.

사용자는 CTA를 보면서 아래를 이해해야 한다.

> 지금 내가 눌러야 할 행동이 무엇이고, 그 행동을 하면 내 경험 정리/이력서 후보/커리어 인사이트가 어떻게 좋아지는지 알 수 있다.

이 산출물의 핵심은 `버튼 존재`가 아니라 `행동 이유 + 기존 화면 연결 + 기대 효과`다.

---

## F-1. CTA 기본 원칙

모든 CTA는 아래 4가지를 가능하면 함께 가진다.

1. 행동 이름
2. 왜 필요한지 설명
3. 기대 효과
4. 기존 화면/handler 연결

예시 구조:

```text
정량 성과 보완하기
최근 기록에는 실행 과정은 잘 남아 있지만 결과 수치가 아직 덜 드러납니다.
처리 건수, 개선률, 사용자 반응을 추가하면 이력서 후보의 설득력이 높아져요.
[성과 정보 추가하기]
```

---

## F-2. 핵심 CTA 목록

이번 작업에서 우선 사용하는 CTA는 아래다.

```text
기록 이어가기
첫 경험 기록하기
이 날짜에 경험 남기기
이력서 후보 보기
커리어 인사이트 보기
성과 정보 추가하기
협업 맥락 보완하기
```

우선순위:

1. `기록 이어가기`
2. `이력서 후보 보기`
3. `성과 정보 추가하기`
4. `협업 맥락 보완하기`
5. `커리어 인사이트 보기`

---

## F-3. CTA별 역할 정의

### F-3-1. 기록 이어가기

목적:

- 사용자가 경험 기록/업무 기록 입력으로 이동한다.
- 기록이 적거나 없는 사용자에게 가장 중요한 CTA다.

권장 카피:

```text
기록 이어가기
```

기록이 없는 경우:

```text
첫 경험 기록하기
```

설명 문구:

```text
오늘 해결한 문제나 협업한 내용을 한 줄만 남겨도 경험 신호를 찾을 수 있어요.
```

연결 후보:

```text
onOpenUpdateView
onOpenAnalysis
기존 기록 추가 handler
jobSidebarView를 resume-update 또는 기록 입력 화면으로 전환하는 기존 흐름
```

새 route를 만들지 않는다.

### F-3-2. 이력서 후보 보기

목적:

- 사용자가 기록 기반 이력서 후보/초안/결과 화면으로 이동한다.
- PASSMAP이 단순 문장 생성기가 아니라, 경험을 이력서 후보로 정리해주는 도구임을 보여준다.

권장 카피:

```text
이력서 후보 보기
```

설명 문구:

```text
최근 기록에서 보이는 경험 신호가 이력서 후보로 어떻게 정리되는지 확인해보세요.
```

연결 후보:

```text
onOpenResumeView
setCurrentScreen("result")
기존 이력서 보기 handler
기존 resume candidate 화면 전환
```

주의:

- 화면 진입 시 AI를 자동 호출하지 않는다.
- 버튼 클릭 시 기존 이력서 보기/후보 흐름으로 연결한다.
- 저장되지 않은 입력이 있는 경우 무리하게 생성하지 않는다.
- 기존 저장 후 생성 흐름이 있으면 사용하고, 없으면 먼저 기록을 남기도록 안내한다.

### F-3-3. 성과 정보 추가하기

목적:

- 기록에는 행동이 있지만 결과/수치가 부족할 때 보완을 유도한다.

권장 카피:

```text
성과 정보 추가하기
```

설명 문구:

```text
처리 건수, 개선률, 사용자 반응을 추가하면 이력서 후보의 설득력이 높아져요.
```

연결:

- 가능하면 기록 보완/기록 입력 화면으로 이동
- 특정 날짜/기록 선택 상태가 있으면 해당 날짜 기록 추가 흐름으로 연결
- 직접 편집 기능이 없으면 일반 기록 입력 CTA로 연결

### F-3-4. 협업 맥락 보완하기

목적:

- 협업 경험은 보이지만 대상/역할/조율 기준이 덜 드러날 때 보완을 유도한다.

권장 카피:

```text
협업 맥락 보완하기
```

설명 문구:

```text
누구와 어떤 기준을 맞췄는지 함께 적으면 커뮤니케이션 역량이 더 분명해져요.
```

연결:

- 기록 입력/보완 흐름으로 연결
- 별도 협업 편집 기능을 새로 만들지 않는다.

## F-4. 상태별 CTA 우선순위

### 기록이 없는 사용자

우선 CTA:

```text
첫 경험 기록하기
```

보조 CTA:

```text
예시 보기
```

단, 예시 보기 기능이 없으면 새로 만들지 않는다. 대신 설명 문구 안에 예시를 넣는다.

### 기록이 1~2개인 사용자

우선 CTA:

```text
기록 이어가기
```

보조 CTA:

```text
이력서 후보 보기
```

설명:

```text
기록을 몇 개 더 남기면 반복 경험과 연결 직무를 더 정확하게 볼 수 있어요.
```

### 기록이 충분한 사용자

우선 CTA:

```text
이력서 후보 보기
```

보조 CTA:

```text
기록 이어가기
```

상황별 CTA:

```text
성과 정보 추가하기
협업 맥락 보완하기
```

## F-5. CTA 디자인 기준

### 버튼 스타일

1차 CTA:

- filled button
- violet/primary 계열
- 카드 안에서 가장 눈에 띄는 버튼

2차 CTA:

- outline 또는 white button
- 1차 CTA보다 낮은 시각적 우선순위

권장 스타일:

```text
rounded-xl px-4 py-2 text-sm font-semibold
```

### 배치

데스크톱:

- 2개 버튼 나란히 배치 가능
- 카드 우측 또는 하단에 배치

모바일:

- 버튼은 1열로 쌓여도 자연스러워야 한다.
- 버튼 텍스트가 줄바꿈되어도 카드 밖으로 넘치면 안 된다.

## F-6. CTA와 설명의 연결

CTA만 단독으로 두지 않는다.

피해야 할 구조:

```text
[기록 이어가기] [이력서 후보 보기]
```

권장 구조:

```text
최근 기록에서 문제 해결과 협업 조율 신호가 보입니다.
이 경험이 이력서 후보에서 어떻게 정리되는지 확인해보세요.
[이력서 후보 보기]
```

---

## F-7. 기능 연결 기준

### 기존 handler 우선

새 기능/새 route를 만들기보다 기존 handler를 재사용한다.

후보:

```text
onOpenResumeView
onOpenUpdateView
onOpenAnalysis
setCurrentScreen("result")
기존 날짜 기반 기록 추가 handler
```

### 기능이 불명확한 경우

기능이 불명확하면 아래 순서로 처리한다.

1. 가장 가까운 기존 화면으로 이동
2. 이동이 어렵다면 CTA를 비활성화하지 말고 일반 기록 입력으로 연결
3. 그래도 불가능하면 CTA를 안내 문구로 바꾸고 최종 보고에 보류로 기록

---

## F-8. 피해야 할 CTA

```text
업데이트 추가
약한 항목 확인
리포트 보기
생성하기
변환하기
```

특히 `이력서로 변환하기`는 서비스가 단순 변환기처럼 보일 수 있으므로 기본값으로 쓰지 않는다. 필요하면 `이력서 후보 보기`를 사용한다.

## F-9. 완료 체크리스트

```text
[ ] 주요 CTA가 `기록 이어가기` / `이력서 후보 보기` 중심으로 정리되었다.
[ ] CTA 주변에 왜 필요한지 설명이 있다.
[ ] CTA가 기존 handler/화면 전환과 연결되어 있다.
[ ] 기록 상태별 CTA 우선순위가 다르게 적용된다.
[ ] 기능이 없는 CTA를 새로 만들지 않았다.
[ ] 화면 진입 시 AI 자동 호출이 없다.
[ ] 모바일에서 CTA가 자연스럽게 줄바꿈/1열 배치된다.
[ ] `이력서로 변환하기`, `업데이트 추가`, `약한 항목 확인` 같은 표현이 주요 CTA로 남지 않았다.
```

## F-10. 검증 방법

### 코드 검증

```powershell
git diff --check
npm run build
```

### 화면 검증

확인 질문:

```text
CTA를 눌렀을 때 기존 화면 흐름이 깨지지 않는가?
CTA가 왜 필요한지 설명되어 있는가?
기록 없음/기록 적음/기록 충분 상태별 CTA가 자연스러운가?
AI 호출이 자동으로 발생하지 않는가?
모바일에서 버튼이 깨지지 않는가?
```

## F-11. 작업 중 판단 규칙

1. 새 기능보다 기존 handler 재사용을 우선한다.
2. CTA에는 이유를 함께 둔다.
3. 사용자의 현재 기록 상태에 맞게 CTA 우선순위를 바꾼다.
4. AI 자동 호출은 하지 않는다.
5. 기능 연결이 불확실하면 보류하고 최종 보고에 남긴다.
6. 산출물 F 체크리스트를 만족하지 못하면 완료로 보고하지 않는다.
