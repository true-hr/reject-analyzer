## Section 1: 목적 (Purpose)

### Axis1 정의

Axis1 "전공과 직무의 연결성"은 **입력한 전공이 목표 직무와 연결될 수 있는 지식/행동 기초를 해석하는 축**입니다.

### 중요한 제한

**Axis1이 평가하는 것:**
- 전공 커리큘럼에서 다루는 지식 영역
- 전공 특성으로 자연스럽게 발달하는 행동 기초
- 직무에서 요구하는 행동과의 개념적 연결성

**Axis1이 평가하지 않는 것:**
- 직무 수행 능력 (Performance capability)
- 취업 또는 합격 가능성 (Employability or hiring probability)
- 산업 이해도 (Industry knowledge) ← Axis2 담당
- 실제 경험이나 프로젝트 성과 (Actual project experience) ← Axis3 담당

### Axis1 점수의 의미

- **direct**: 전공과 직무의 개념적 연결이 분명함
- **adjacent**: 전공과 직무의 일부 개념이 맞닿음
- **weak**: 전공과 직무 간 개념적 거리가 있음
- **mismatch**: 전공과 직무 간 개념적 연결이 약함

## Section 3: 현재 로직 계약 (Current Logic Contract)

### 점수 책임

- **점수 계산 파일**: `src/data/transitionLite/newgradAxis1MajorPriorRegistry.js`
- **점수 구조**: 30개 전공 × 14개 직무 대분류 matrix
- **점수 밴드**: direct / adjacent / weak / mismatch (이미 존재)
- **오버라이드**: override / exceptionAdjustment 지원

### Axis1이 보존해야 할 것

| 항목 | 유지 여부 | 설명 |
|---|---|---|
| 점수 matrix (30×14) | ✅ 변경 금지 | 기존 로직 유지 |
| direct/adjacent/weak/mismatch 밴드 | ✅ 변경 금지 | 기존 분류 체계 유지 |
| `classifyTransition.js` 점수 로직 | ✅ 변경 금지 | 다운스트림 계약 영향 |
| `rawScore`, `displayScore`, `band` 필드 | ✅ 변경 금지 | API 계약 |

### Axis1이 주의할 것

- **경험을 점수의 중심으로 삼지 않기**: Axis3 역할과 혼동 금지
  - ❌ "인턴십 경험으로 점수 조정"
  - ✅ "전공 기초만으로 판단, 경험은 보충 설명"
  
- **산업 이해도를 혼합하지 않기**: Axis2 역할과 혼동 금지
  - ❌ "이 산업에서 필요한 지식"
  - ✅ "이 직무에서 중요한 행동"

## Section 4: 세부판독 목표 구조 (Detail Reading Target Structure)

### 기본 템플릿

```
현재 입력한 전공만 보면 {targetJobLabel} 직무와의 연결은 {bandPhrase} 편입니다.

{majorLabel} 전공은 {targetJobCategoryLabel}에서 중요한 {jobCoreActions} 중 
{majorRelatedActions}와는 연결될 수 있지만, 현재 입력만으로는 {missingActions}까지 
직접 드러나지는 않습니다.

이 연결을 더 강하게 보려면, 전공 수업이나 프로젝트 안에서 {followUpQuestion}이 
있었는지 함께 떠올려보는 것이 좋습니다.
```

### 필드 정의

| 필드 | 생성 방식 | 예시 |
|---|---|---|
| **targetJobLabel** | 입력받은 값 | "백엔드 개발" |
| **bandPhrase** | direct/adjacent/weak/mismatch 규칙에 따라 변환 | "비교적 분명한" |
| **majorLabel** | 입력받은 전공 | "컴퓨터공학" |
| **targetJobCategoryLabel** | 직무 대분류 라벨 | "개발 (엔지니어링)" |
| **jobCoreActions** | jobCategoryCoreActionsRegistry에서 조회 | "논리적 설계, 시스템 구현, 디버깅" |
| **majorRelatedActions** | canonicalActions ∩ coreActions (의미 기반) | "논리적 사고, 시스템 구현" |
| **missingActions** | coreActions - majorRelatedActions (설명하기 어려운 것 1~2개) | "실제 사용자 피드백에 따른 수정" |
| **followUpQuestion** | 전공 학습 근거 회상 가이드 | "알고리즘 과제나 시스템 프로젝트에서 성능 최적화를 다룬 경험" |

## Section 5: bandPhrase 규칙 (Band to Phrase Conversion)

Axis1 점수 밴드를 사용자 문장으로 변환하는 규칙입니다.

| 밴드 | bandPhrase | 사용 문맥 | 금지 표현 |
|---|---|---|---|
| **direct** | "비교적 분명한" | 전공과 직무 개념이 명확하게 맞닿음 | "직접 맞닿는", "핵심 과업과 잘 연결되는", "전공 적합성이 유리한" |
| **adjacent** | "일부 보이는" | 전공과 직무 개념의 일부가 맞닿음 | "관련 신호가 있는", "일부 맞닿아 있는" |
| **weak** | "아직 약한" | 전공과 직무 간 거리가 있음 | "부족한", "불리한", "더 구체화하면 좋은" |
| **mismatch** | "현재 입력만으로는 약한" | 전공과 직무의 개념적 연결이 약함 | "직무 수행 가능성이 낮은", "합격 가능성이 낮은" |

## Section 6: 필요한 보조 Registry 설계 (Auxiliary Registry Design)

### 6.1 jobCategoryCoreActionsRegistry

**목적**: 14개 직무 대분류별 핵심 행동 언어 제공

**데이터 소스**: 기존 `job ontology`의 `families[].strongSignals`, `mediumSignals` 참고

**필드 설계**:

```javascript
{
  category: "MARKETING",           // 직무 대분류 코드
  label: "마케팅",                 // 한글 라벨
  coreActions: [
    "고객 반응 분석",
    "콘텐츠와 메시지 개선",
    "채널 운영",
    "캠페인 성과 확인"
  ],
  caution: "세부 직무에 따라 브랜드, 퍼포먼스, 콘텐츠 중심 행동이 달라질 수 있음"
}
```

**초기 구성**: 14개 대분류 (MARKETING, SALES, SERVICE_PLANNING, DATA_ANALYSIS 등)

**구현 시점**: 추후 (이번 문서는 계약만)

### 6.2 majorCanonicalActionsRegistry

**목적**: 30개 전공별 canonicalActions 제공 (전공 학습에서 자연스럽게 연결되는 행동)

**필드 설계**:

```javascript
{
  majorKey: "COMPUTER_SCIENCE",    // 전공 코드
  label: "컴퓨터공학",              // 한글 라벨
  canonicalActions: [
    "논리적 문제 분석",
    "알고리즘 사고",
    "시스템 구현",
    "데이터 처리"
  ],
  learningBasis: [
    "프로그래밍 과제",
    "자료구조/알고리즘 수업",
    "시스템 또는 데이터 관련 프로젝트"
  ]
}
```

**초기 구성**: 30개 전공 (COMPUTER_SCIENCE, BUSINESS_ADMINISTRATION, DESIGN 등)

**구현 시점**: 추후 (이번 문서는 계약만)

## Section 7: majorRelatedActions 생성 원칙 (Generation Principle)

### 원칙

1. **문자열 교집합 금지**: 단어 매칭이 아니라 **의미 범주 기반** 연결
2. **대표 연결 행동만**: 420개 모두 매핑하지 않음. 1차 버전은 **1~2개 대표만** 선정
3. **낮은 확신 처리**:
   - 확신 높음: "연결될 수 있습니다" (단정)
   - 확신 낮음: "전공 수업/프로젝트에서 어떤 방식으로 다뤘는지 확인이 필요합니다" (유보)

### 예시

| 전공 | 직무 대분류 | canonicalActions | coreActions | majorRelatedActions | 근거 |
|---|---|---|---|---|---|
| 컴퓨터공학 | 개발 (엔지니어링) | 논리적 분석, 알고리즘, 시스템 구현 | 논리적 설계, 시스템 구현, 디버깅 | "논리적 설계", "시스템 구현" | 프로그래밍 기초와 직접 연결 |
| 경영학 | 서비스기획 | 프로세스 개선, 수치 분석, 사람 이해 | 고객 니즈 파악, 프로세스 설계, 성과 분석 | "프로세스 설계", "성과 분석" | 경영학의 조직/프로세스 이론과 연결 |

## Section 8: missingActions 생성 원칙 (Missing Actions Principle)

### 원칙

1. **부족함을 단정하지 않기**: "능력이 부족합니다" 금지
2. **현재 입력 제한 유지**: "현재 입력만으로는 아직 드러나지 않습니다" 표현 필수
3. **대표 action 1~2개만**: 모든 부족함을 나열하지 않음
4. **선정 기준**: 직무 핵심 행동 중 **전공만으로 설명하기 가장 어려운 것**

### 금지 예시

❌ "마케팅 성과 분석 능력이 부족합니다"

❌ "산업 경험이 없어서 불리합니다"

### 허용 예시

✅ "현재 입력만으로는 고객 반응을 보고 콘텐츠나 메시지를 실제로 바꾼 장면까지는 드러나지 않습니다"

✅ "전공 수업 범위에서는 마케팅 캠페인의 성과를 측정하고 개선하는 과정을 직접 다루기 어렵습니다"

## Section 9: followUpQuestion 생성 원칙 (Follow-up Question Principle)

### 원칙

1. **자소서 작성법 금지**: "자소서에 이렇게 쓰세요" 안 됨
2. **기억 회상 가이드**: "이런 장면이 있었는지 떠올려보세요"
3. **학습 근거 범주 사용**:
   - 전공 수업 (e.g., "알고리즘 수업", "마케팅 관론")
   - 팀 프로젝트 (e.g., "팀 프로젝트에서 역할")
   - 졸업작품 (e.g., "졸업작품에서")
   - 공모전 (e.g., "공모전 참여에서")
   - 실험/데이터 분석 (e.g., "실험 설계나 데이터 분석")
   - 사용자 조사 (e.g., "사용자 조사나 피드백 수집")
   - 운영 개선 (e.g., "동아리/학생회 운영 개선")

4. **입력받지 않은 사실 단정 금지**: UI에서 수집하지 않은 정보 가정 금지

### 금지 예시

❌ "자소서에서는 이런 프로젝트를 먼저 언급하세요"

❌ "당신의 인턴십 경험이 있나요?" (이미 입력받은 경우는 제외)

❌ "아마 당신도 OO 경험이 있을 겁니다" (가정)

### 허용 예시

✅ "전공 수업이나 팀 프로젝트에서 데이터를 수집하고 분석한 경험이 있었는지 떠올려보세요"

✅ "시스템 프로젝트에서 코드 성능을 개선한 경험이 있었다면, 그것이 근거가 될 수 있습니다"

✅ "공모전이나 학생회 활동에서 팀을 주도한 경험이 있었는지 확인해보세요"

## Section 10: 금지 표현 (Forbidden Phrases)

아래 표현과 유사 표현은 **절대 사용 금지**입니다.

| 금지 표현 | 이유 | 허용 대안 |
|---|---|---|
| "직접 맞닿습니다" | Axis1이 개념 연결성만 평가 | "연결은 비교적 분명한 편입니다" |
| "핵심 과업과 잘 연결됩니다" | 수행 능력 단정 | "중요한 행동과 연결될 수 있습니다" |
| "전공 적합성이 유리합니다" | 합격 판정 | "전공 기초가 도움이 될 수 있습니다" |
| "전공 mismatch가 불리합니다" | 부정적 단정 | "현재 입력만으로는 약한 편입니다" |
| "직무 수행 가능성이 높습니다" | Axis1 범위 초과 | 사용 불가 |
| "합격 가능성이 높습니다" | Axis1 범위 초과 | 사용 불가 |
| "전공 역량이 있습니다" | 역량 평가는 Axis3 | "전공 기초가 있습니다" (제한적 사용) |
| "산업 경험이 있습니다" | Axis2 영역 | 사용 불가 |
| "관련 신호가 있습니다" | 모호한 추상 표현 | "연결될 수 있습니다" |
| "일부 맞닿아 있습니다" | 모호한 추상 표현 | "일부 보이는 편입니다" |
| "더 구체화하면 좋습니다" | 자소서 지도 | "어떤 장면이 있었는지 확인이 필요합니다" |
| "채용자는 이렇게 볼 것입니다" | 채용사 판단 추측 | 사용 불가 |
| "자소서에는 이렇게 쓰세요" | 자소서 지도 | 사용 불가 |
| "이력서에서는 이렇게 적으세요" | 이력서 지도 | 사용 불가 |

## Section 11: 허용 표현 (Allowed Phrases)

Axis1 세부판독에서 **반드시 사용**해야 할 표현입니다.

| 허용 표현 | 사용 맥락 | 예시 |
|---|---|---|
| "현재 입력한 전공만 보면" | 세부판독 문장 시작 | "현재 입력한 전공만 보면 XX 직무와의 연결은 ..." |
| "연결은 비교적 분명한 편입니다" | direct 밴드 설명 | bandPhrase for direct |
| "연결은 일부 보이는 편입니다" | adjacent 밴드 설명 | bandPhrase for adjacent |
| "연결은 아직 약한 편입니다" | weak 밴드 설명 | bandPhrase for weak |
| "현재 입력만으로는 약한 편입니다" | mismatch 밴드 설명 | bandPhrase for mismatch |
