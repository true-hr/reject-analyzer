# Home First Implementation Plan

## 문서 목적
PASSMAP의 실제 첫 제작 대상을 홈 화면으로 잠그기 위한 실행 우선순위 문서.

화면 설계와 별도로, 무엇을 먼저 만들고 무엇을 나중에 미룰지 판단 기준을 정리한다.

## 홈을 우선 제작하는 이유
- 홈은 전체 기능을 연결하는 실제 진입 허브다.
- 홈을 먼저 만들면 이후 각 기능 화면을 독립 제작하더라도 연결 흐름이 흔들리지 않는다.
- 미팅 데모용 설명 화면을 넘어 실제 제품 첫 구현 기준 화면으로 삼을 수 있다.

## 전체 화면 우선순위
| 우선순위 | 화면 | 이유 |
| --- | --- | --- |
| P0-1 | 홈 | 전체 상태, 활동, 액션을 묶는 첫 구현 기준 |
| P0-2 | 업무 업데이트 | 홈의 최근 활동 데이터 소스 |
| P0-3 | 분석 리포트 | 홈 KPI와 최근 리포트 연결 핵심 |
| P1-1 | JD 준비도 | 홈 KPI 상세 진입 화면 |
| P1-2 | 이력서 관리 | 홈 최신화율 카드 연결 화면 |
| P1-3 | 목표 설정 | 홈 현황 패널의 핵심 정보 원천 |
| P1-4 | 내 프로필 | 개인 설정/상태 보강 |
| P1-5 | 액션 가이드 독립화 | 추천 액션의 후속 상세 화면 |

## 홈 최소 제작 범위
- 사이드바
  - 전체 정보구조의 허브로 동작해야 한다.
  - 이번 라운드에서는 메뉴 노출과 활성 상태만 우선 구현한다.
  - 세부 권한 제어와 개인화는 아직 만들지 않는다.
- KPI 카드
  - 홈의 상태 요약 핵심이므로 반드시 포함한다.
  - 카드명, 수치, 보조문구, 클릭 이동만 우선 구현한다.
  - 상세 필터와 기간 선택은 지금 만들지 않는다.
- 현황 패널
  - 사용자의 현재 진행 상태를 한 문장 수준으로 보여준다.
  - 목표, 최근 업데이트, 최근 분석 상태, 핵심 과제까지 표시한다.
  - 복잡한 편집 기능은 지금 넣지 않는다.
- 성과 패널
  - 홈이 운영형 제품처럼 보이게 만드는 핵심 요약 블록이다.
  - 누적 수치와 성과 요약만 우선 노출한다.
  - 세부 성과 리포트 drill-down은 다음 단계로 미룬다.
- 최근 업무 업데이트
  - 최근 활동 허브 역할을 위해 필요하다.
  - 최근 3건 리스트와 보기/수정/분석 연결 액션까지만 구현한다.
  - 정렬/검색/무한스크롤은 지금 만들지 않는다.
- 최근 분석 리포트
  - 분석 기능이 홈과 실제로 연결되어 보이게 해야 한다.
  - 최근 3건과 적합도 표시, 바로가기 버튼만 우선 구현한다.
  - 리포트 비교 기능은 제외한다.
- 추천 액션
  - 다음 행동 허브 역할을 담당한다.
  - 추천 카드 2~3개와 기본 CTA만 우선 구현한다.
  - 개인화 로직 고도화는 이후 단계로 미룬다.
- 활동 추이 그래프
  - 홈의 운영감과 제품 인상을 강화한다.
  - 최근 8주 기준 더미 또는 연결 가능한 실제 데이터로 우선 구현한다.
  - 고급 인터랙션은 지금 만들지 않는다.

## 다음 제작 순서
1. 홈 골격 레이아웃 고정
2. KPI 카드와 현황/성과 패널 연결
3. 최근 업무 업데이트, 최근 분석 리포트 연결
4. 추천 액션과 활동 추이 그래프 추가
5. 업무 업데이트 화면 상세화
6. 분석 리포트 화면 상세화
7. JD 준비도, 이력서 관리, 목표 설정 순차 연결

## 구현 전 잠글 결정사항
- 홈을 앱의 기본 랜딩 화면으로 둘지 여부
- KPI 카드 4개 명칭과 수치 계산 기준
- 최근 리스트의 기본 표시 개수
- 추천 액션 데이터가 더미인지 실제 계산값인지
- 활동 추이 그래프 데이터 소스

## 홈 제작 완료 기준
- 사이드바와 핵심 4개 KPI 카드가 렌더링된다.
- 현황, 성과, 최근 업데이트, 최근 리포트, 추천 액션, 그래프가 한 화면에 정리된다.
- 각 블록에 실제 또는 명시적 placeholder 이동 경로가 있다.
- 데모 설명과 실제 구현 브리프로 동시에 사용할 수 있다.

## 이후 코드 작업 브리프 초안
- 목표: 홈을 PASSMAP 첫 구현 기준 화면으로 고정한다.
- 필수 구성: 사이드바, KPI 4카드, 현황, 성과, 최근 업데이트, 최근 리포트, 추천 액션, 활동 추이 그래프.
- 비필수 구성: 고급 필터, 개인화 추천 고도화, 세부 드릴다운.
- 구현 원칙: 과장 없이 placeholder와 실제 연결 범위를 구분한다.

## JOB Shell-Level Rail 진행 상태
- 해결 완료
  - left rail이 centered wrapper 바깥의 Shell level로 이동했다.
  - right content width bottleneck이 해소되어 `work` 화면은 wide dashboard 구조를 자연스럽게 사용할 수 있게 됐다.
- 확인된 원인
  - Shell의 `mx-auto + max-w-6xl` anchor 때문에 JOB 내부 2열 패치만으로는 폭 문제를 근본 해결할 수 없었다.
- 다음 작업
  - `analysis` view에만 inner wrapper max-width를 다시 도입해 landing 밀도를 회복한다.
  - `work` view는 현재 wide 상태를 유지한다.
  - `resume` view는 현 상태를 유지하거나 중간 폭 정책을 별도로 검토한다.

## Current Priority Update
- 현재 우선순위는 정교한 엔진 확장이 아니라 기록 단위 분리와 시간 흐름 복원이다.
- P0-1: `오늘 기록` / `이번 주 기록` 입력 단위를 분리한다.
- P0-2: `업무 관리`, `이력서 쉽게 업데이트 하기` 두 화면에 달력 또는 날짜 기반 상태 UI를 넣는다.
- P0-3: `업무 관리` 상단의 운영 숫자형 KPI를 행동/시간 기반 정보로 교체한다.

## 우선순위에서 내린 것
- `분석 리포트 수`, `JD 비교 수` 같은 운영 숫자는 현재 주연 정보로 두지 않는다.
- 우측 설명 박스 확장, 미세 카드 구성, 시각적 디테일 조정보다 시간 축과 기록 상태 노출을 먼저 만든다.
- 엔진 정교화는 위 세 가지 구조가 먼저 고정된 뒤에 다시 올린다.

## 진행 완료
- Shell-level left rail 도입 완료.
- right content width bottleneck 제거 완료.
- `work` 화면의 1주 strip을 월간 캘린더로 전환 완료.
- `work` 캘린더를 normalized records + derived summary 구조로 전환 완료.
- 선택 날짜 자산화 카드와 이번 달 자산 요약 반영 완료.
- 결과적으로 `work`는 기록 -> 해석 -> 자산화 흐름이 날짜 중심으로 보이는 운영 대시보드 단계까지 올라왔다.

## 남은 작업
- 다음 작업은 자산화 결과를 실제 반영 후보 카드로 연결하는 것이다.
- 이번 달 반영 후보 문장, 포트폴리오 bullet 후보, 재사용 가치가 높은 자산을 `work`에서 바로 보이게 해야 한다.
- reflected sentence를 단순 결과 문장이 아니라 이력서 / 포트폴리오 반영 후보 영역으로 승격할 필요가 있다.
- Notion / Google Calendar 실제 adapter 연결 전까지는 현재 source contract를 유지한 상태로 consumer만 확장한다.

## 주의사항
- UTF-8 postcheck는 일부 라운드에서 콘솔 출력 문제와 함께 보류 이슈가 있었다. 문서 본문과 콘솔 노이즈를 분리해 확인해야 한다.
- `src/App.jsx`는 여전히 큰 orchestration 파일이라 대형 diff를 피해야 한다.
- 레이아웃/앵커 문제가 의심되면 내부 카드 패치로 오래 끌지 말고 Shell/anchor 수준 root cause를 먼저 확인한다.
